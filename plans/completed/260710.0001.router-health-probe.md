# Task Breakdown

## Overview

Add an **active router reachability/credential probe** to the `yarddogd` query daemon's
`GET /health/check`. The probe logs into the GPON router's web UI and logs out again — it
proves "the daemon can still authenticate against the router" **without ever rebooting** —
and reports as one more entry in the existing flat `services` map of the health response
(the user will name it `gipon`).

The design must resolve a real architectural tension: **the daemon deliberately reads no
router credentials today.** `LoadDaemonConfig` requires only `DAEMON_TOKEN`, and a
monitor-only host is expected to run *just* the daemon with no router password on disk
(`cmd/yarddogd/main.go` doc: "it never reads the router password or the Telegram DSN, and
never fails to start over their absence"). We keep that property intact by making the
router probe **entirely optional**: the daemon reads router config only if it is present,
wires the probe only when a full set of router credentials is configured, and otherwise
starts exactly as it does today with `DAEMON_TOKEN` alone.

The probe is **live on every `/health/check` request** — it is a real-time check of whether
the app can authenticate against the router *right now*, so it performs an actual
login/logout per request rather than serving a cached verdict (operator decision:
`/health/check` reflects current state, not a stale window). To keep a burst of concurrent
health requests from stampeding the Nokia's session table, concurrent `CheckUP` calls are
**coalesced into a single in-flight login** (a stdlib single-flight: parallel callers share
one login/logout and its result); this bounds *concurrency* to one open session without
adding any time-based staleness. The one operational boundary this leaves — documented, not
enforced — is that a monitor polling `/health/check` at sub-minute cadence authenticates
against the router at that cadence; the endpoint is token-gated and LAN-only, so the
operator controls the poll rate.

### Fit with the layered architecture

The dependency rule stays `domain ← services ← {infrastructure, gateway/*} ← cmd/*`. The
probe adds no new package and no DTO change. New code lands as:

- **`gateway/router/nokia.go`** — a new exported driver method `CheckCredentials(ctx)` (or
  `Ping(ctx)`): login then logout, no reboot. This is the "add a capability to the existing
  device driver" move, not a new device — the device-driver-family rule (a new *device* is a
  driver file + factory case) is untouched.
- **`gateway/router/`** — a new **health-probe adapter** (`credential_probe.go`) that wraps
  a `Credentialer` + a `services.Clock` and implements `services.HealthProbe` (`Name()` +
  `CheckUP(ctx)`), with the TTL cache living inside it. It carries the
  `var _ services.HealthProbe = (*credentialProbe)(nil)` assertion.
- **`services/ports.go`** — a new consumer-side port, `Credentialer`, with the single method
  `CheckCredentials(ctx) error`. Defined at the consumer (the probe adapter, in
  `gateway/router`, is the caller — but the *port* lives in `services` per "ports live at
  the consumer", since the adapter imports `services` to satisfy `HealthProbe` anyway).
- **`infrastructure/daemon_config.go`** — `LoadDaemonConfig` gains four **optional** router
  keys and an all-or-nothing validation; `DaemonConfig` gains the resolved router fields +
  the probe display name + the TTL.
- **`cmd/yarddogd/main.go`** — wire the router driver + credential probe into the
  `services.NewInspector(...)` variadic **only when** router config is present.

No change to `services/health.go` (the `Inspector` stays per-dependency-logic-free — we add
a dependency by adding a probe, exactly as the inspector pattern demands), no change to
`gateway/dto/api.go`, no change to `gateway/httpapi/*` (the flat `services` map already
renders any probe the inspector holds).

## Locked Decisions (supersede any task detail below that conflicts)

Confirmed with the operator; final for this implementation:

1. **Variant A — the daemon probes the router live.** The daemon holds an optional
   `services.Credentialer` and performs the login itself. `/health/check` is the operator's
   single "is the whole app OK right now?" view — metrics stay in `/api/v1/metrics`, and the
   reboot stays the collector's already-shipped background job; **the daemon never reboots.**
2. **No time cache — live on every request.** DROP the TTL entirely: remove
   `YARDDOG_DAEMON_HEALTH_ROUTER_TTL` (Task 4) and the `ttl`/`checkedAt`/cache fields
   (Task 3). Each `CheckUP` performs a real login/logout.
3. **Single-flight coalescing instead of a cache.** Concurrent `CheckUP` calls share one
   in-flight login (stdlib only: a `sync.Mutex` guarding a shared
   `*call{done chan struct{}; err error}` — the classic single-flight, ~15 lines; do NOT
   import `x/sync`). A caller that finds a login in flight waits on its channel and returns
   that shared result; when none is in flight, the next call starts a fresh one. Bounds
   *concurrent* sessions to one, with zero time-staleness. A healthy result is plain `"ok"`
   (no age annotation — it is always live).
4. **`Credentialer` stays separate from `Rebooter`** (Task 1) — the daemon can check the
   login but is structurally incapable of rebooting.
5. **Flat `services` entry**, display name configurable via `YARDDOG_DAEMON_HEALTH_ROUTER`,
   default `router` (operator sets `gipon`).
6. **Logout = local cookie-jar reset** (no invented `/logout.cgi`); a reset/logout failure
   never fails the probe.
7. **Docs + wiring (Task 5/6):** note the endpoint authenticates against the router on every
   hit (token-gated + LAN-only, so the operator controls poll rate). Register the router
   probe **last** in the inspector list and keep `CHECK_TIMEOUT` < `DAEMON_HEALTH_TIMEOUT`
   (Risk R3) so a wedged router login cannot starve the other probes.

Open Questions Q1 (TTL) and Q2 (cache age) are resolved by decision 2/3 (no cache); Q3
logout = jar-reset; Q4 = opt-in per host; Q5 name default = `router`.

## Assumptions

- **The Nokia driver's login contract is the only one available.** `NokiaDriver.login()`
  today treats "session cookie (`sid`/`lsid`) present in the jar after POST `login.cgi`" as
  the sole success signal (`gateway/router/nokia.go` L117–147; design risk §14 — the
  contract is inferred, not documented). The credential probe reuses exactly this signal;
  it does not invent a stronger check.
- **There is no documented logout endpoint.** `Reboot` deliberately never logs out ("the
  session dies with the reboot", L74–77), so no `/logout.cgi` path is proven to exist. The
  probe's "logout" is therefore best-effort and its failure must NOT fail the probe (see
  Task 2 + Risk R1 + Open Question Q3). The probe's job is "can I authenticate?"; a clean
  logout is hygiene, not the assertion.
- **The collector may or may not share a host with the daemon.** On a full watchdog host
  both run and the router creds are already on disk (in the shared env file); on a
  monitor-only host only the daemon runs and no router creds exist. The optional-probe
  design serves both without a second config surface. See Open Question Q4.
- **`DB_PATH` and the env file are already shared** between the two binaries; this plan adds
  no new file, only new keys read by the daemon's existing loader.
- **Go 1.22+ / stdlib-only + `modernc.org/sqlite`** constraints are unchanged; the probe
  and its cache use only `net/http`, `sync`, `context`, `time`.

## Tasks

### Task 1: Add the `Credentialer` port in `services`

- **Description:** Define a new consumer-side port in `services/ports.go`:

  ```go
  // Credentialer proves the daemon can still authenticate against the router
  // web UI without rebooting it: CheckCredentials logs in and logs out,
  // returning nil on a successful login. gateway/router's device drivers are
  // the production implementations; the credential health probe is the sole
  // consumer. Kept separate from Rebooter so a probe can never reboot: the
  // two capabilities are different verbs on the same device.
  type Credentialer interface {
      CheckCredentials(ctx context.Context) error
  }
  ```

  Place it next to `Rebooter` (the sibling device verb) in file-declaration order.
  Rationale for a *new* port rather than extending `Rebooter`: interface segregation keeps
  the reboot capability un-reachable from the read-only daemon by construction — the daemon
  never holds a `Rebooter`, only a `Credentialer` — so a health probe physically cannot
  trigger a reboot. This also honours "ports live at the consumer" and "no port for
  symmetry": the port earns its place because a real consumer (the probe) needs exactly this
  one method and nothing wider.

- **Acceptance Criteria:**
  - [ ] `services.Credentialer` exists with the single method
    `CheckCredentials(ctx context.Context) error`.
  - [ ] Godoc starts with the identifier name, ends with a period, explains the *why*
    (separate from `Rebooter` so the daemon cannot reboot).
  - [ ] `go build ./...` and `go vet ./...` pass; no import added to `services` beyond the
    existing `context`.
- **Pitfalls & edge cases:** Do NOT add this method to `Rebooter` — that would let the
  daemon's composition root hold a value that can reboot. Do NOT define the port in
  `gateway/router` next to the impl (that violates "ports live at the consumer"). Keep the
  method signature `error`-only; the probe, not the port, owns caching/naming.
- **Complexity:** Easy

### Task 2: Add `CheckCredentials` to `NokiaDriver` (login + best-effort logout, no reboot)

- **Description:** Add an exported method to `gateway/router/nokia.go` that reuses the
  existing unexported `login(ctx)` and then attempts a logout, never rebooting:

  ```go
  // CheckCredentials proves the configured credentials still authenticate
  // against the router web UI: it performs the same login.cgi POST as Reboot
  // (success = a session cookie was set) and then best-effort logs the session
  // out, so a health probe polling this never reboots the router. login
  // failure is the only failure it reports; a logout error is swallowed (the
  // session will expire on its own, and there is no documented logout contract
  // — design §14), so a router that authenticates but rejects the logout path
  // is still reported healthy. Implements services.Credentialer.
  func (d *NokiaDriver) CheckCredentials(ctx context.Context) error {
      if err := d.login(ctx); err != nil {
          return fmt.Errorf("router: check credentials: %w", err)
      }
      d.logout(ctx) // best-effort; see doc — a logout failure is not a probe failure
      return nil
  }
  ```

  Add the `var _ services.Credentialer = (*NokiaDriver)(nil)` assertion next to the existing
  `var _ services.Rebooter = (*NokiaDriver)(nil)`.

  For `logout`: the safe MVP is a **cookie-jar reset** (drop the session cookie locally so
  the driver's next `login` starts clean) rather than hitting an unproven `/logout.cgi`.
  This matches the "do not invent bytes / endpoints" discipline in `gateway/dto/nokia.go`
  L8–9. If a real logout endpoint is later confirmed from the router's own admin UI, it
  slots into `logout` behind the same best-effort contract. **This is Open Question Q3 —
  confirm before implementing whether to (a) reset the jar only, or (b) POST a logout path.**

  Because the swallowed logout error would otherwise trip the "no skipped errors" rule,
  handle it explicitly — assign and log, or reset the jar (which returns no error) — never
  `_ = d.logout(ctx)` silently.

- **Acceptance Criteria:**
  - [ ] `NokiaDriver.CheckCredentials(ctx)` returns nil when `login` succeeds (a session
    cookie is set), and a wrapped `ErrLoginFailed` (via `%w`) when it does not.
  - [ ] `CheckCredentials` **never** issues a `reboot.cgi` request — verified by a test that
    fails if the `httptest` server ever sees `/reboot.cgi`.
  - [ ] A logout/session-teardown failure does **not** make `CheckCredentials` return an
    error (test: server 500s the logout path, or jar-reset path, probe still returns nil).
  - [ ] `var _ services.Credentialer = (*NokiaDriver)(nil)` compiles.
  - [ ] No `_`-discarded error anywhere in the new code.
  - [ ] Table/subtest layout: one `TestNokiaDriver_CheckCredentials` with subtests
    (`"login ok returns nil"`, `"bad credentials returns ErrLoginFailed"`, `"never calls
    reboot.cgi"`, `"logout failure is swallowed"`), each driven by an `httptest.Server` — no
    real network, no real sleep.
- **Pitfalls & edge cases:**
  - The cookiejar is shared with `Reboot`; if `logout` resets it, ensure a later `Reboot`
    on the same driver still works (in practice the daemon never holds a `Rebooter`, so this
    is only a concern if a single driver instance is reused for both — the collector builds
    its own).
  - `login` drains and closes its response body already; the logout path must do the same
    (`io.Copy(io.Discard, resp.Body)` + `defer resp.Body.Close()`) if it makes an HTTP call,
    or nothing if it only resets the jar.
  - Respect `ctx` in any HTTP call (`http.NewRequestWithContext`) so the whole-sweep deadline
    from the `Inspector` can cancel it — a router mid-reboot must not hang the health sweep.
  - Do not change `Reboot`'s existing behaviour or its "no logout after reboot" comment.
- **Complexity:** Medium

### Task 3: The TTL-caching credential probe adapter in `gateway/router`

- **Description:** Add `gateway/router/credential_probe.go`: a `services.HealthProbe`
  wrapping a `services.Credentialer`, a `services.Clock`, a configurable display name, and a
  TTL. It performs a real `CheckCredentials` at most once per TTL window and serves the
  cached verdict (annotated with its age) in between. Concurrency-safe (the daemon serves
  concurrent `/health/check` requests).

  ```go
  // NewCredentialProbe builds a credentialProbe that reports router
  // authentication health under name, calling cred.CheckCredentials at most
  // once per ttl and serving the cached verdict (with its age) in between.
  // clock lets tests drive the TTL deterministically; production passes
  // SystemClock. A ttl <= 0 disables caching (every sweep probes live) —
  // config validation should keep ttl positive.
  func NewCredentialProbe(name string, cred services.Credentialer, clock services.Clock, ttl time.Duration) *credentialProbe

  type credentialProbe struct {
      name  string
      cred  services.Credentialer
      clock services.Clock
      ttl   time.Duration

      mu        sync.Mutex
      checkedAt time.Time // zero until the first probe
      lastErr   error     // nil = healthy; the cached verdict
      cached    bool
  }

  var _ services.HealthProbe = (*credentialProbe)(nil)
  ```

  `CheckUP(ctx)` logic:
  1. Lock; if `cached` and `now.Sub(checkedAt) <= ttl`, return the cached verdict — but if
     it was an error, wrap it so the report shows the age, e.g.
     `fmt.Errorf("%w (cached %s ago)", lastErr, age.Round(time.Second))`. On a cached
     *success* return `nil` (the report just shows `"ok"`; age is only interesting when the
     annotation carries signal — Open Question Q2 asks whether to surface cache age on the
     healthy path too).
  2. Otherwise call `p.cred.CheckCredentials(ctx)`, store `checkedAt = now`, `lastErr =
     err`, `cached = true`, and return `err` (freshly, no age annotation).

  Hold the mutex across the live `CheckCredentials` call so two concurrent sweeps do not both
  hit the router (the second blocks, then sees the fresh cached result). This serializes
  health sweeps against the router, which is exactly the desired cost bound. Because the
  `Inspector` already runs probes sequentially under one shared deadline, the mutex only ever
  contends across *separate concurrent HTTP requests*, not within one sweep.

  `Name()` returns the injected `name` (so the operator sets `gipon`), not a hardcoded
  string — this is the configurable-display-name requirement.

- **Acceptance Criteria:**
  - [ ] `credentialProbe` implements `services.HealthProbe`; assertion compiles.
  - [ ] `Name()` returns the injected name verbatim.
  - [ ] First `CheckUP` calls the underlying `Credentialer` exactly once; a second `CheckUP`
    within `ttl` (fake clock advanced `< ttl`) calls it **zero** more times and returns the
    same verdict.
  - [ ] A `CheckUP` after the fake clock advances **past** `ttl` calls the `Credentialer`
    again (cache expiry).
  - [ ] A cached *error* verdict is surfaced with its age (`"… (cached Ns ago)"`); a cached
    *success* returns nil.
  - [ ] Concurrency: N goroutines calling `CheckUP` simultaneously on a cold cache invoke the
    underlying `Credentialer` **once** (guarded by a call-counter fake; run under
    `make test-race` locally). No data race on `checkedAt`/`lastErr`/`cached`.
  - [ ] Tests use a fake `Credentialer` (call counter + programmable error) and a fake
    `Clock` (both in `credential_probe_test.go`, each with its
    `var _ services.X = (*fake)(nil)` assertion) — no real router, no real sleep.
  - [ ] One `TestCredentialProbe_CheckUP` with subtests + one `TestCredentialProbe_Name`.
- **Pitfalls & edge cases:**
  - **Do not** hold the lock while formatting/logging outside the critical section beyond
    what's needed; but DO hold it across the live call (that is the point). A `context`
    cancel mid-call still returns through `CheckCredentials`; store that error as the cached
    verdict like any other (a cancelled probe is "unhealthy this window").
  - A `ttl <= 0` must not divide-by-zero or panic; treat it as "always probe live". Config
    validation (Task 4) keeps it positive, but the adapter must be safe regardless.
  - Use `services.Clock.Now()` for *all* time reads (never `time.Now()`), or the fake-clock
    tests are meaningless and the TTL races the wall clock.
  - The cached error must be wrapped with `%w` so `errors.Is` still works if a caller ever
    needs it, and so the age annotation reads naturally.
  - Keep the probe read-only w.r.t. the router beyond the single login (+ best-effort
    logout) — no retries-in-a-loop, no reboot, ever.
- **Complexity:** Medium

### Task 4: Optional router config in `LoadDaemonConfig` (all-or-nothing)

- **Description:** Extend `infrastructure/daemon_config.go` so the daemon *optionally* reads
  router credentials. New keys, **all optional**, read via the existing bare-key lookup
  (the loader prepends `YARDDOG_`; the operator sets the prefixed form):

  | Logical key (code) | Env var (operator) | Default | Meaning |
  |---|---|---|---|
  | `ROUTER_ADDR` | `YARDDOG_ROUTER_ADDR` | `http://192.168.1.1` | router web-UI base URL (shared vocabulary with the collector) |
  | `ROUTER_USER` | `YARDDOG_ROUTER_USER` | *(none)* | router login user — presence is what *enables* the probe |
  | `ROUTER_PASS` | `YARDDOG_ROUTER_PASS` | *(none)* | router login password |
  | `ROUTER_KIND` | `YARDDOG_ROUTER_KIND` | `nokia` | device driver family (shared with the collector) |
  | `DAEMON_HEALTH_ROUTER` | `YARDDOG_DAEMON_HEALTH_ROUTER` | `router` | the probe's display name in `services` (user sets `gipon`) |
  | `CHECK_TIMEOUT` | `YARDDOG_CHECK_TIMEOUT` | `5s` | per-request HTTP timeout for the driver (reuse the collector's key/default) |

  (The TTL key from the earlier draft is removed — see Locked Decisions 2/3: no cache.)

  **Enablement + all-or-nothing rule:** the probe is enabled iff **both** `ROUTER_USER` and
  `ROUTER_PASS` are set (non-empty). `ROUTER_ADDR`/`ROUTER_KIND`/name/TTL all have safe
  defaults, so they are never "required". If **exactly one** of `ROUTER_USER` / `ROUTER_PASS`
  is set, **fail fast** (exit 1) with a message naming the missing one — a half-configured
  router probe is an operator error, not a silent no-op:

  ```
  router health probe partially configured: YARDDOG_ROUTER_USER set but YARDDOG_ROUTER_PASS missing (set both to enable, neither to disable)
  ```

  Add a `RouterHealth *DaemonRouterConfig` (nil when the probe is disabled) to `DaemonConfig`,
  or equivalently a `RouterHealthEnabled bool` + fields — prefer the nil-pointer form so
  "disabled" is unrepresentable-as-half-set at the type level:

  ```go
  type DaemonConfig struct {
      // … existing fields …
      RouterHealth *DaemonRouterHealth // nil = router probe disabled
  }

  // DaemonRouterHealth is the resolved, validated router-probe config; it is
  // non-nil only when both ROUTER_USER and ROUTER_PASS were supplied.
  type DaemonRouterHealth struct {
      Kind        domain.RouterKind
      Addr        string
      User        string
      Pass        string
      ProbeName   string
      TTL         time.Duration
      HTTPTimeout time.Duration
  }
  ```

  Parse `ROUTER_KIND` via `domain.ParseRouterKind` (reject an unknown kind at load, exit 1,
  matching the collector). Parse the TTL and HTTP timeout via `time.ParseDuration` with the
  same error-message shape the loader already uses. Reject `TTL <= 0` at load (an operator
  setting `0m` almost certainly meant "disable" — tell them to unset it instead).

- **Acceptance Criteria:**
  - [ ] With neither `YARDDOG_ROUTER_USER` nor `YARDDOG_ROUTER_PASS` set, `LoadDaemonConfig`
    returns a config with `RouterHealth == nil` and **no error** — the daemon still starts on
    `DAEMON_TOKEN` alone (the property this whole plan protects).
  - [ ] With both set, `RouterHealth` is non-nil, fully populated, with defaults applied for
    the absent optional keys.
  - [ ] With exactly one of user/pass set, `LoadDaemonConfig` returns an error naming the
    missing key; the daemon exits 1.
  - [ ] An unparseable `ROUTER_KIND`, `DAEMON_HEALTH_ROUTER_TTL`, or `CHECK_TIMEOUT`, or a
    non-positive TTL, returns an error (exit 1).
  - [ ] `DAEMON_TOKEN` remains the sole *unconditionally* required key (existing test still
    passes unchanged).
  - [ ] One `TestLoadDaemonConfig` gains subtests: `"router probe disabled when creds
    absent"`, `"router probe enabled when both creds present"`, `"partial router creds
    fail"`, `"invalid router kind fails"`, `"invalid ttl fails"`, `"non-positive ttl fails"`,
    `"router defaults applied"`. Use `t.Setenv` under the `envPrefix` namespace like the
    existing config tests.
- **Pitfalls & edge cases:**
  - `getOrDefault` treats `KEY=` (empty) the same as unset — so an operator who writes
    `YARDDOG_ROUTER_PASS=` gets "disabled/half-set", consistent with the rest of the loader.
    Test both "unset" and "empty" for the enablement check.
  - Do **not** make `ROUTER_ADDR` required — it has a default; requiring it would break the
    "monitor-only host, no router config" path.
  - Do **not** scrub the router password from memory here (the loader already documents why:
    the shared env file puts it in the environment regardless; `daemon_config.go` L11–17).
  - Keep the `YARDDOG_` prefix out of the code literals — pass bare keys to `get`, exactly
    like the existing `DAEMON_*` reads.
  - Reuse the collector's `CHECK_TIMEOUT` key rather than inventing a daemon-specific one;
    the driver's request timeout means the same thing in both binaries. (Alternative:
    `DAEMON_ROUTER_TIMEOUT`; rejected to avoid a redundant key — see Trade-off T5.)
- **Complexity:** Medium

### Task 5: Wire the probe into the daemon composition root (conditionally)

- **Description:** In `cmd/yarddogd/main.go`, after building `clk` and before
  `services.NewInspector(...)`, build the router driver + credential probe **only when**
  `cfg.RouterHealth != nil`, and append it to the inspector's variadic probe list:

  ```go
  clk := infrastructure.SystemClock{}

  probes := []services.HealthProbe{
      infrastructure.NewFreshnessProbe(st, clk, cfg.StaleAfter),
  }
  if rh := cfg.RouterHealth; rh != nil {
      driver, err := router.New(rh.Kind, router.RouterConfig{
          Addr:    rh.Addr,
          User:    rh.User,
          Pass:    rh.Pass,
          Timeout: rh.HTTPTimeout,
      })
      if err != nil {
          log.Printf("yarddogd: router health probe: %v", err)
          return domain.ExitConfigError
      }
      probes = append(probes, router.NewCredentialProbe(rh.ProbeName, driver, clk, rh.TTL))
  }

  insp := services.NewInspector(cfg.HealthTimeout, append([]services.HealthProbe{st}, probes...)...)
  srv := httpapi.NewServer(services.NewQueryService(st), insp, cfg.Token, version, clk.Now())
  ```

  Note `router.New` returns a `services.Rebooter`; the probe needs a `services.Credentialer`.
  Two clean options — **decide before implementing (relates to Q3/architecture):**
  - **(a)** Have `router.New` also return a value usable as a `Credentialer`. Since the
    concrete `*NokiaDriver` satisfies both interfaces, the simplest is a second, sibling
    factory `router.NewCredentialer(kind, cfg) (services.Credentialer, error)` that dispatches
    the same way — symmetric with `New`, honours the device-family rule (add a device = a
    case in *both* tiny factories), and keeps `main` from type-asserting.
  - **(b)** Type-assert the `Rebooter` returned by `New` to `services.Credentialer` in
    `main`. Rejected: an assertion in the composition root is fragile and couples `main` to
    the knowledge that the concrete driver implements both.

  **Recommend (a):** a `NewCredentialer` factory beside `New`. It is the device-family
  pattern applied honestly (the factory dispatches `ROUTER_KIND`), and it keeps `main`
  dependency-injection-only.

  A build error / bad `ROUTER_ADDR` at this point is a **configuration error → exit 1**
  (`domain.ExitConfigError`), consistent with the existing config-error paths in `run()`.

- **Acceptance Criteria:**
  - [ ] When `cfg.RouterHealth == nil`, the inspector holds exactly the two existing probes
    (`sqlite` + `collector-freshness`); the daemon starts and `/health/check` is unchanged.
  - [ ] When `cfg.RouterHealth != nil`, the inspector additionally holds the router probe
    under the configured name; a `/health/check` hit shows `services["gipon"]` (or whatever
    name) = `"ok"` on success, the verbatim login error otherwise.
  - [ ] A router driver build failure (unparseable `ROUTER_ADDR`) makes `run()` return
    `domain.ExitConfigError` (1) with a logged reason.
  - [ ] The daemon's exit-code contract (0 clean / 1 config / 2 listen) is unchanged — the
    router probe never introduces a new exit code.
  - [ ] `resolveEnvPath` and the rest of `run()` are untouched.
- **Pitfalls & edge cases:**
  - The `st` (`*Store`) probe must stay first/last consistently — it already is a
    `HealthProbe` (the `sqlite` probe); keep it in the list exactly once. Don't accidentally
    drop it while restructuring into a slice.
  - `router` is a new import in `cmd/yarddogd/main.go` (`gateway/router`) — that is allowed:
    a `cmd/*` main may import any `gateway/*`. Confirm no import cycle (there can't be — router
    imports `services`, not `cmd`).
  - Do not build the driver unconditionally "to keep it simple" — that reintroduces the
    router-creds requirement the plan is designed to avoid.
- **Complexity:** Easy

### Task 6: Docs — README env table + `yarddog.env.example` + CLAUDE.md health section

## Risks

- **R1 — the logout contract is inferred, like the login one.** There is no documented
  `/logout.cgi`; the safe MVP resets the local cookie jar instead of hitting an unproven
  endpoint. If we later POST a real logout path and the router 500s it, the swallow-on-error
  contract (Task 2) keeps the probe healthy — but the *login* signal itself is already
  inferred (`gateway/router/nokia.go` L117–121, design §14), so a router firmware change to
  the login response could turn the probe falsely red. Mitigation: the probe only asserts
  what `Reboot` already asserts; no new fragility beyond the existing login inference.
- **R2 — session-table pressure / collision with the collector's reboot login.** Even TTL-
  bounded, the probe opens a session every ~10m. If the Nokia caps concurrent sessions
  low and the collector reboots at the same moment, one login could evict the other.
  Mitigation: TTL default 10m (tunable up), the probe logs out (frees the session), and the
  probe and collector are different processes on their own cookie jars. Residual: a
  simultaneous reboot+probe race is possible but harmless (worst case one login fails and the
  probe reports one red window). Open Question Q1 sets the default TTL.
- **R3 — a wedged router login starves later probes in the sweep.** The `Inspector` runs
  probes sequentially under one shared `DAEMON_HEALTH_TIMEOUT` (default 5s); a router that
  accepts the TCP connection but never responds could burn most of that budget before the
  `CHECK_TIMEOUT` (5s) fires — leaving little for probes after it. Mitigation: keep
  `CHECK_TIMEOUT` **< `DAEMON_HEALTH_TIMEOUT`** so the driver's own timeout trips first, and
  register the router probe **last** in the list so a wedge starves nothing downstream. This
  is exactly the "first independent, non-SQLite probe is when to revisit sequential probing"
  note in CLAUDE.md — flag it, do not re-architect the inspector now (Trade-off T3).
- **R4 — password now flows through the daemon on a monitor-only host that opted in.** The
  moment an operator sets `YARDDOG_ROUTER_PASS` for the daemon, the "daemon needs no router
  secret" property is *voluntarily* traded for the probe. That is the intended, explicit
  trade — but the docs (Task 6) must make it clear that enabling the probe puts the router
  password in the daemon's environment. The daemon still never *reboots*, so the blast radius
  of a compromised daemon is "can read the router login page", not "can reboot the router"
  (because it holds a `Credentialer`, not a `Rebooter`).

## Trade-offs

- **T1 — Flat `services` map entry, not a nested `devices` group.** *Chosen:* add the probe
  as `services["gipon"] = "ok"` in the existing `map[string]string`, no DTO change. *Rejected:*
  a nested `{"devices": {...}}` block in `HealthResponse`. Reason: the inspector pattern's
  core rule is "add a dependency by adding a probe, not by touching the aggregator/DTO"; a
  nested group would change `gateway/dto/api.go`, `handleHealth`, and every existing
  consumer's parsing for zero functional gain. A router is just another dependency, exactly
  like `sqlite` and `collector-freshness`. If the daemon ever probes *many* heterogeneous
  devices this can be revisited, but one optional probe does not earn a schema change.
- **T2 — Optional probe via nil-pointer config, not a separate daemon binary or always-on
  requirement.** *Chosen:* one `DaemonConfig.RouterHealth *DaemonRouterHealth`, nil = off.
  *Rejected:* (a) making router creds required for the daemon (breaks monitor-only hosts,
  contradicts `cmd/yarddogd/main.go`'s own doc); (b) a third binary for "watchdog+router
  health" (violates "no speculative structure / no third binary until a real consumer needs
  it"). The nil-pointer makes "half-configured" unrepresentable and keeps the single daemon.
- **T3 — TTL cache in the probe adapter, not in the `Inspector` or a shared cache layer.**
  *Chosen:* the `credentialProbe` owns its own mutex + cache. *Rejected:* teaching the
  `Inspector` to cache (it must stay per-dependency-logic-free — CLAUDE.md), or a generic
  caching-probe decorator (speculative; only one probe needs a cache today). Cost-bounding is
  a property *of this dependency*, so it lives in this probe, exactly as the health-check
  skill puts the 2–3h cache inside the paid-API probe.
- **T4 — New `Credentialer` port, not an extension of `Rebooter`.** *Chosen:* a one-method
  port the daemon holds. *Rejected:* adding `CheckCredentials` to `Rebooter`. Reason: interface
  segregation makes it *impossible* for the read-only daemon to reboot (it never holds a
  `Rebooter`), which is a security property, not just tidiness. Cost: one more tiny interface
  and (recommended) a sibling `NewCredentialer` factory — cheap, and it follows the
  device-family pattern already in place.
- **T5 — Reuse `CHECK_TIMEOUT` for the driver's per-request timeout, not a new daemon key.**
  *Chosen:* the daemon reads the same `CHECK_TIMEOUT` the collector uses. *Rejected:* a
  `DAEMON_ROUTER_TIMEOUT`. Reason: it means the same thing (bound one router HTTP request) and
  a redundant key is config surface for no benefit. Residual: on a monitor-only host that sets
  neither, the default (5s) applies — fine, and it stays `< DAEMON_HEALTH_TIMEOUT` by default.
- **T6 — Best-effort logout = local jar reset by default, not an unproven `/logout.cgi`.**
  *Chosen:* drop the session cookie locally so we never invent an endpoint. *Rejected:* POST a
  guessed logout path. Reason: `gateway/dto/nokia.go` already codifies "do not invent bytes";
  the session expires on its own, and the probe's assertion is "can I authenticate", for which
  a clean logout is hygiene, not proof. Upgradable behind the same best-effort contract if a
  real endpoint is confirmed (Open Question Q3).

## Open Questions (need the user's decision before/at implementation)

- **Q1 — Default TTL value.** Plan assumes `YARDDOG_DAEMON_HEALTH_ROUTER_TTL = 10m`. Too
  short → more router logins; too long → a stale "router unreachable" verdict lingers. Is
  10m right, or do you want 5m / 15m / 30m? (Router session-table size is the real
  constraint.)
- **Q2 — Surface cache age on the *healthy* path too?** Plan surfaces age only on a cached
  *error* (`"… (cached Ns ago)"`) and returns plain `"ok"` when healthy-and-cached. Do you
  want the healthy entry to also show freshness (e.g. `"ok (checked 3m ago)"`) so an operator
  can tell a live probe from a cached one? That is a small string change but it alters the
  `services` value format for this entry.
- **Q3 — Logout mechanism.** Default is a **local cookie-jar reset** (never touches an
  unproven endpoint). Do you know of a real `/logout.cgi`-style path on your Nokia G-240W-A
  (from the router's own admin UI network capture)? If yes, provide the exact path/body and
  we POST it (still best-effort); if not, we reset the jar.
- **Q4 — Enable the probe on the full-watchdog host, or only monitor-only?** On a host where
  the collector already runs, the router creds are on disk anyway — do you want the daemon's
  router probe on *there* (redundant-ish with the collector's own reboot login proving the
  same creds), or only on a **monitor-only** host where nothing else touches the router? The
  probe is opt-in either way; this just sets your default env on each host class.
- **Q5 — Probe display name default.** Plan defaults `YARDDOG_DAEMON_HEALTH_ROUTER = router`
  and you override to `gipon`. Fine to ship `router` as the default, or would you rather the
  default be `gipon` / `gpon` / `ont`? (Purely the fallback when the key is unset.)
