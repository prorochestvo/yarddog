# Task Breakdown

## Overview

Add a second, **long-running** binary — `yarddogd`, the **query daemon** — that serves the
data the collector records (runs, checks, and the 003 host telemetry) over a read-only
**JSON REST HTTP API** on the LAN. The existing cron collector becomes `cmd/yarddog`; the
daemon is `cmd/yarddogd`. Both share the same layered packages (`domain`, `services`,
`infrastructure`, `gateway/*`) and the same SQLite file; each has its own composition root
under `cmd/`. This is the first thing yarddog has ever done that opens a listening port, so
the plan also rewrites the two CLAUDE.md/README sections that assert "no listening ports /
health checks N/A".

The daemon is deliberately **read-mostly**: the collector (a separate process) is the sole
writer; the daemon opens the same WAL database and only reads (its one write is the
idempotent `CREATE TABLE IF NOT EXISTS` migration, safe to run from either process). The API
is guarded by a **LAN bind + a shared token** header; a dumb `/ping` liveness probe stays
ungated, and a real `/health/check` readiness probe (the inspector pattern) sits behind the
token.

### Fit with the layered architecture

The dependency rule stays `domain ← services ← {infrastructure, gateway} ← main`, and this
plan adds a **second `main`** rather than a shared bootstrap layer (CLAUDE.md Code
Organization Principles: "a second binary gives it its own composition root under `cmd/`").
New code lands as:

- **`gateway/httpapi/`** — a new **inbound** HTTP adapter (mirrors the existing outbound
  `gateway/telegram`/`gateway/router` idiom): a `Server` over a stdlib `http.ServeMux`, the
  handlers, the token middleware, domain→DTO mapping. Imports `services` + `domain` +
  `gateway/dto`. **Never** imports `infrastructure`.
- **`gateway/dto/`** — the JSON response shapes (pure structs, json tags), alongside the
  existing Telegram/Nokia wire shapes.
- **`services/query.go`** — the consumer-defined **read port** (`HistoryRepository`) + a
  thin `QueryService` (limit clamping, not-found translation, run+checks assembly) +
  `MetricsFilter`. Imports `domain` only.
- **`services/health.go`** — the `HealthProbe` port + the `Inspector` aggregator (the
  inspector pattern). Imports `domain` + stdlib only.
- **`infrastructure/`** — read methods on `Store` (`LatestHost`, `LatestMetrics`,
  `ListMetrics`, `ListRuns`, `RunByID`) implementing `HistoryRepository`; the two health
  probes (`Store.CheckUP` + a `freshnessProbe`) implementing `HealthProbe`; a connection-pool
  parameter (`OpenStore`); and `LoadDaemonConfig`.
- **`cmd/yarddog/`** — the moved collector (`main.go` + `main_test.go`, unchanged behaviour).
- **`cmd/yarddogd/`** — the new daemon composition root: config, signal-driven graceful
  shutdown, wiring, `http.Server` with timeouts.

`net/http`'s `ServeMux` (Go 1.22+ method-and-path patterns: `mux.HandleFunc("GET
/api/v1/runs/{id}", …)` + `r.PathValue("id")`; the module is `go 1.26.4`) means **zero new
dependencies** — no router library, no web framework. `modernc.org/sqlite` stays the only
third-party import.

## Assumptions

- **003 (host telemetry) is merged before 004 starts.** 004 *serves* 003's `metrics`
  (EAV) and `host` tables and reuses 003's `domain.MetricSample`/`HostInfo` value types and
  the `metrics`/`host` column consts in `infrastructure/store.go`. 004 is planned against
  003's final tree and rebases on it. If 003 slips, the `runs`/`checks` endpoints, the store
  health probe, and the whole scaffold still ship; only `/api/v1/host` and
  `/api/v1/metrics*` (and the freshness probe's exact query) block on 003. See Risk R7.
- **The daemon is read-only over the DB except migration.** It never writes `runs`,
  `checks`, `metrics`, `host`, or `tg_outbox`. Its only write is the idempotent migration,
  which is safe to run from both processes (`CREATE TABLE IF NOT EXISTS`). Whoever starts
  first creates the schema; on a fresh box the daemon returns empty results (or 404 for the
  singleton "latest" endpoints), never an error.
- **LAN + shared token is the whole security model.** The daemon binds a LAN address and
  requires an `X-Yarddog-Token` header (constant-time compared) on every route except
  `/ping`. There is no TLS, no per-user auth, no rate limiting — the router UI it manages is
  already plain HTTP on the LAN, so the trust boundary is "anyone on the LAN". This is
  documented, not accidental (README LAN-only warning + firewall guidance).
- **`DAEMON_TOKEN` is required; empty ⇒ fail fast, exit 1.** A daemon with no token would
  serve the internal topology (dependency names, error strings, run history) to the whole
  LAN unauthenticated. `LoadDaemonConfig` refuses to start without it.
- **The daemon needs no `flock`.** The cron collector uses `flock` to prevent overlapping
  runs; the daemon's single-instance guarantee is the port bind itself — a second instance
  fails with `EADDRINUSE` (exit 2). Do not add lock bookkeeping.
- **Times over the wire are UTC RFC3339 strings.** Same format the store persists. Nullable
  timestamps (`*time.Time`) serialize as JSON `null` (`*string`), not `""`.
- **The daemon's exit codes are its own contract, not the collector's.** `0` clean
  shutdown, `1` config error, `2` listen/serve error. The collector's 0–5 reboot-centric
  table does not apply to a server (documented in the daemon's package doc + README).
- Module path `github.com/prorochestvo/yarddog`, `go 1.26.4`; completion gate stays
  `CGO_ENABLED=0 go vet ./... && CGO_ENABLED=0 go test ./...`; every build is `CGO_ENABLED=0`
  and both binaries cross-compile to `linux/arm64`, `linux/amd64`, `darwin/arm64`.

## API surface (routes, auth, responses)

Version the path (`/api/v1`). All responses are JSON; all times UTC RFC3339. `/ping` is
ungated; every other route requires the `X-Yarddog-Token` header.

| Method + path | Auth | Query params | Success | Error codes |
|---|---|---|---|---|
| `GET /ping` | none | — | `200` `{"status":"ok"}` | — |
| `GET /health/check` | token | — | `200` report (healthy) / **`503`** report (unhealthy) | `401` |
| `GET /api/v1/host` | token | — | `200` `HostResponse` | `401`, `404` (no data yet), `500` |
| `GET /api/v1/metrics/latest` | token | — | `200` `MetricsLatestResponse` | `401`, `404` (no data yet), `500` |
| `GET /api/v1/metrics` | token | `since` (RFC3339), `collector` (one of the 7), `limit` (int) | `200` `MetricsListResponse` | `400` (bad param), `401`, `500` |
| `GET /api/v1/runs` | token | `limit` (int) | `200` `RunsListResponse` | `400`, `401`, `500` |
| `GET /api/v1/runs/{id}` | token | — | `200` `RunDetailResponse` | `400` (bad id), `401`, `404`, `500` |

**List vs singleton on an empty DB.** List endpoints (`/metrics`, `/runs`) return `200` with
`count: 0` and an empty array — an empty list is a valid answer. The singleton "latest"
endpoints (`/host`, `/metrics/latest`) return `404 {"error":"no data"}` when no row exists
yet, so a client can tell "the collector has never run" apart from "here is the latest".

**Limits.** `/api/v1/runs`: `limit` default 50, max 500. `/api/v1/metrics`: `limit` default
100, max 1000. `limit<=0` or absent ⇒ default; `limit>max` ⇒ clamped to max (not an error).
An unparseable `limit`/`since`/`collector` ⇒ `400`.

### Response DTO shapes (`gateway/dto`, pure json-tagged structs)

```jsonc
// PingResponse
{ "status": "ok" }

// HealthResponse  (200 when status=true, 503 when false)
{
  "status": true,
  "server":   { "version": "v0.2.0", "uptime": "2h34m12s" },
  "services": { "sqlite": "ok", "collector-freshness": "ok" }
}
// unhealthy example (HTTP 503):
// "services": { "sqlite": "ok",
//               "collector-freshness": "last run 3h12m0s ago exceeds 1h30m0s staleness budget" }

// HostResponse
{ "run_id": 128, "ts": "2026-07-07T04:07:00Z", "hostname": "pi5", "os": "linux", "arch": "arm64" }

// MetricDTO  (value is null + ok=false for an unavailable sample)
{ "run_id": 128, "ts": "2026-07-07T04:07:00Z", "collector": "temperature",
  "name": "cpu-thermal", "value": 52.35, "unit": "celsius", "ok": true, "error": "" }

// MetricsLatestResponse  (all metric rows of the newest run that has metrics)
{ "run_id": 128, "ts": "2026-07-07T04:07:00Z", "count": 15, "metrics": [ /* MetricDTO… */ ] }

// MetricsListResponse  (filtered history; each row carries its own run_id/ts)
{ "count": 42, "metrics": [ /* MetricDTO… */ ] }

// RunDTO  (mirrors domain.Run; nullable *At fields are JSON null when unset)
{ "id": 128, "started_at": "2026-07-07T04:07:00Z", "mode": "soft", "internet_ok": true,
  "action": "none", "reboot_started_at": null, "router_down_at": null, "router_up_at": null,
  "internet_restored_at": null, "finished_at": "2026-07-07T04:07:03Z",
  "outcome": "ok", "error": "" }

// CheckDTO  (latency_ms null when the probe recorded none)
{ "run_id": 128, "ts": "2026-07-07T04:07:00Z", "phase": "initial", "target": "1.1.1.1:443",
  "kind": "ip", "ok": true, "latency_ms": 12, "error": "" }

// RunsListResponse
{ "count": 10, "runs": [ /* RunDTO… */ ] }

// RunDetailResponse
{ "run": { /* RunDTO */ }, "checks": [ /* CheckDTO… */ ] }

// ErrorResponse  (every 4xx/5xx body)
{ "error": "run not found" }
```

Field types: `value` → `*float64`, `latency_ms` → `*int64`, `internet_ok` → `*bool`, all
nullable timestamps → `*string` (RFC3339). `error`/`outcome`/`mode`/`action` are plain
strings (empty string when absent). The **structs live in `gateway/dto`** with no internal
imports (mirroring `dto.TelegramSendMessageRequest`); the **domain→DTO mapping** (including
time formatting) lives in `gateway/httpapi`, exactly as `gateway/telegram` fills the Telegram
DTO from its own fields.

## Query port (`services.HistoryRepository`, implemented by `*infrastructure.Store`)

Consumer-defined in `services/query.go`; `infrastructure.Store` satisfies it. Reuses the two
read methods `Store` already exposes off-port today (`ListChecksByRun`, and `RunByID` wraps
the existing `GetRun`), and adds four.

```go
// services/query.go
type MetricsFilter struct {
    Since     time.Time       // zero value = no lower bound
    Collector domain.Collector // "" = all collectors
    Limit     int             // <=0 = service default; clamped to max in QueryService
}

type HistoryRepository interface {
    LatestHost(ctx context.Context) (domain.HostRecord, bool, error)          // newest host row; ok=false if none
    LatestMetrics(ctx context.Context) ([]domain.MetricRecord, error)         // all metric rows of the newest run that has metrics; empty if none
    ListMetrics(ctx context.Context, f MetricsFilter) ([]domain.MetricRecord, error)
    ListRuns(ctx context.Context, limit int) ([]domain.Run, error)            // newest first
    RunByID(ctx context.Context, id int64) (domain.Run, bool, error)          // ok=false = not found (no sql.ErrNoRows leak)
    ListChecksByRun(ctx context.Context, runID int64) ([]domain.Check, error) // already implemented on Store
}
```

`domain.HostRecord`/`domain.MetricRecord` are new read types (a persisted row = 003's value
type + its `run_id`/`ts` keys), analogous to how `domain.Run`/`domain.Check` are the shapes
`RunRepository` returns:

```go
// domain/history.go  (new; depends on 003's MetricSample/HostInfo)
type HostRecord   struct { RunID int64; TS time.Time; Host domain.HostInfo }
type MetricRecord struct { RunID int64; TS time.Time; Sample domain.MetricSample }
```

`QueryService` is the thin application layer between the handlers and the port — it holds the
cross-cutting read logic so handlers stay pure marshalling:

```go
// services/query.go
type QueryService struct{ repo HistoryRepository }
func NewQueryService(repo HistoryRepository) *QueryService { return &QueryService{repo: repo} }

func (q *QueryService) Runs(ctx context.Context, limit int) ([]domain.Run, error) {
    return q.repo.ListRuns(ctx, clampLimit(limit, defaultRunsLimit, maxRunsLimit))
}
func (q *QueryService) Metrics(ctx context.Context, f MetricsFilter) ([]domain.MetricRecord, error) {
    f.Limit = clampLimit(f.Limit, defaultMetricsLimit, maxMetricsLimit)
    return q.repo.ListMetrics(ctx, f)
}
func (q *QueryService) Run(ctx context.Context, id int64) (domain.Run, []domain.Check, bool, error) {
    run, found, err := q.repo.RunByID(ctx, id)
    if err != nil || !found {
        return domain.Run{}, nil, found, err
    }
    checks, err := q.repo.ListChecksByRun(ctx, id)
    return run, checks, true, err
}
// LatestHost / LatestMetrics delegate straight through.
```

## Health endpoints (the inspector pattern — health-check skill, applied exactly)

Two endpoints, two jobs. Both mandatory.

- **`/ping` (liveness)** — ungated, always `200 {"status":"ok"}`, touches **no** dependency.
- **`/health/check` (readiness)** — token-gated; runs an `Inspector` that probes every
  dependency under **one whole-sweep `context.WithTimeout`**, isolated (one dead/slow probe
  never hangs the endpoint or hides the others; each probe is `recover`-wrapped), read-only.
  Response `{status, server{version,uptime}, services{name→"ok"|error}}`; **HTTP `200` when
  `status=true`, `503` when `status=false`.**

The `HealthProbe` port + `Inspector` live in `services/health.go` (so infrastructure probes
can carry the `var _ services.HealthProbe = (*T)(nil)` assertion — `infrastructure→services`
is legal, `infrastructure→gateway` is not). `gateway/httpapi` consumes the concrete
`*services.Inspector` and renders the report.

```go
// services/health.go
type HealthProbe interface {
    Name() string                       // stable label in the report, e.g. "sqlite"
    CheckUP(ctx context.Context) error  // nil = healthy; error text shown verbatim
}
type Inspector struct{ probes []HealthProbe; timeout time.Duration }
func NewInspector(timeout time.Duration, probes ...HealthProbe) *Inspector { … }
func (i *Inspector) CheckUp(ctx context.Context) (healthy bool, report map[string]string) {
    ctx, cancel := context.WithTimeout(ctx, i.timeout)
    defer cancel()
    healthy, report = true, make(map[string]string, len(i.probes))
    for _, p := range i.probes {
        name := p.Name()
        if name == "" { continue }
        if err := probeSafely(ctx, p); err != nil { // recover-wrapped
            report[name] = err.Error(); healthy = false; continue
        }
        report[name] = "ok"
    }
    return healthy, report
}
```

**The two probes (in `infrastructure`, each owning its `Name()`+`CheckUP`):**

1. **`sqlite`** — `Store.CheckUP` does `db.PingContext(ctx)` **and** `SELECT 1` (ping alone
   can pass while queries fail).
2. **`collector-freshness`** — a `freshnessProbe` that reads the newest `runs.started_at`
   and marks the service unhealthy if it is older than `DAEMON_STALE_AFTER` (⇒ the cron
   collector has stopped writing ⇒ the data the daemon serves is stale). No runs at all ⇒
   unhealthy ("no runs recorded yet"). Holds a `services.Clock` so tests are deterministic.

## Config split per binary

Keep one shared env mechanism (`loadEnvFile` + a `newEnvLookup` helper giving the real-env-
wins-over-file lookup closure), two loaders, each validating only what its binary needs.

| | Collector (`LoadConfig`, unchanged) | Daemon (`LoadDaemonConfig`, new) |
|---|---|---|
| Required | `LABEL`, `TELEGRAMBOT_DSN`, `ROUTER_USER`, `ROUTER_PASS` | `DAEMON_TOKEN` |
| Reads | `ROUTER_*`, `CHECK_*`, `RECOVERY_*`, `REBOOT_*`, `METRICS_*`, `DB_PATH`, `RETENTION_DAYS` | `DAEMON_ADDR`, `DB_PATH`, timeouts, `DAEMON_STALE_AFTER`, `DAEMON_MAX_CONNS` |
| Never reads | — | router creds, Telegram DSN, check targets |

The daemon must **not** be forced through the collector's required-key validation (it has no
router password). New daemon env vars:

| Variable | Default | Meaning |
|---|---|---|
| `DAEMON_TOKEN` | — (**required**) | shared secret in the `X-Yarddog-Token` header; empty ⇒ exit 1 |
| `DAEMON_ADDR` | `127.0.0.1:8420` | bind address; set to a **LAN** IP:port in production (loopback default is fail-safe) |
| `DB_PATH` | `/var/lib/yarddog/yarddog.db` | shared SQLite file (same as the collector) |
| `DAEMON_STALE_AFTER` | `90m` | `/health/check` marks the collector dead if the newest run is older than this (≈3× the 30-min soft cron) |
| `DAEMON_READ_TIMEOUT` | `10s` | `http.Server.ReadTimeout` |
| `DAEMON_WRITE_TIMEOUT` | `15s` | `http.Server.WriteTimeout` |
| `DAEMON_IDLE_TIMEOUT` | `60s` | `http.Server.IdleTimeout` |
| `DAEMON_HEALTH_TIMEOUT` | `5s` | inspector whole-sweep budget |
| `DAEMON_MAX_CONNS` | `2` | SQLite read-pool size (see Trade-off T5) |

### Task 9: `cmd/yarddogd` — the daemon composition root

- **Description:** Replace the Task 1 placeholder with the real daemon: config → store (read
  pool) → query service → probes → inspector → HTTP server → graceful shutdown.
  - `cmd/yarddogd/main.go` (`package main`; `// Command yarddogd …` doc naming the LAN/token
    model and the 0/1/2 exit codes; `var version = "dev"`):
    ```go
    func main() { os.Exit(run()) }
    func run() int {
        envPath := flag.String("env", "", "path to the env file (default /etc/yarddog.env, ./.env)")
        flag.Parse()
        cfg, err := infrastructure.LoadDaemonConfig(resolveEnvPath(*envPath))
        if err != nil { log.Printf("yarddogd: config error: %v", err); return domain.ExitConfigError }

        ctx, stop := signal.NotifyContext(context.Background(), syscall.SIGINT, syscall.SIGTERM)
        defer stop()

        st, err := infrastructure.OpenStore(ctx, cfg.DBPath, cfg.MaxConns)
        if err != nil { log.Printf("yarddogd: open store: %v", err); return domain.ExitConfigError }
        defer func() { if err := st.Close(); err != nil { log.Printf("yarddogd: close store: %v", err) } }()

        clk := infrastructure.SystemClock{}
        insp := services.NewInspector(cfg.HealthTimeout, st, infrastructure.NewFreshnessProbe(st, clk, cfg.StaleAfter))
        srv := httpapi.NewServer(services.NewQueryService(st), insp, cfg.Token, version, clk.Now())

        httpServer := &http.Server{
            Addr: cfg.Addr, Handler: srv,
            ReadTimeout: cfg.ReadTimeout, WriteTimeout: cfg.WriteTimeout, IdleTimeout: cfg.IdleTimeout,
            ReadHeaderTimeout: cfg.ReadTimeout,
        }
        return serve(ctx, httpServer) // ListenAndServe in a goroutine; on ctx.Done → Shutdown(10s)
    }
    ```
  - `serve` runs `ListenAndServe` in a goroutine, returns `exitListenError` (2) if it fails
    with anything but `http.ErrServerClosed`, and on `ctx.Done()` calls `Shutdown` under a
    bounded context, returning `domain.ExitOK`. `resolveEnvPath` is copied from the collector
    (trivial, per-binary — not shared).
  - `main_test.go`: unit-test `resolveEnvPath` and `serve`'s shutdown path (start on
    `127.0.0.1:0`, hit `/ping`, cancel ctx, assert clean return) — no external network.
- **Acceptance Criteria:**
  - [ ] `LoadDaemonConfig` failure (e.g. missing `DAEMON_TOKEN`) ⇒ exit `1`, no listener opened.
  - [ ] A bind failure (e.g. an in-use `127.0.0.1:0`… use an already-bound port) ⇒ exit `2`.
  - [ ] `serve` test: server answers `/ping` `200`, then a `SIGINT`-equivalent ctx cancel
        drains and returns `0` within the shutdown budget.
  - [ ] No `flock` and no router/Telegram wiring in the daemon; `make build` builds both
        binaries; `make build-arm`/`make build-mac` build both; gate green.
  - [ ] `var _ = domain.ExitOK` etc. — the daemon reuses `domain.ExitOK`/`ExitConfigError` and
        a local `exitListenError = 2`.
- **Pitfalls & edge cases:**
  - `signal.NotifyContext` is the whole shutdown trigger — do not add a manual `signal.Notify`
    channel on top. `defer stop()` restores default signal handling.
  - `ListenAndServe` returns `http.ErrServerClosed` on a clean `Shutdown` — treat that as
    success (exit 0), any other error as exit 2. Classic footgun: reporting `ErrServerClosed`
    as a failure.
  - The daemon runs the idempotent migration via `OpenStore`→`migrate`. On a fresh box where
    the collector has never run, this creates empty tables; the API then returns empty/`404`,
    which is correct — do not treat "no data" as an error at startup.
  - Bound `Shutdown` with its own `context.WithTimeout` (e.g. 10s) so a stuck in-flight request
    cannot hang termination; systemd will `SIGKILL` after its own `TimeoutStopSec` anyway.
  - The daemon and collector must run as users that can both read/write the DB's `-wal`/`-shm`
    files (chmod 0600 by the store). Simplest: same user; else same group with group-read.
    Flag this in the docs task, not enforced in code.
- **Complexity:** Medium.

## Risks

- **R1 — The daemon serves the internal topology to the LAN.** `/health/check` and error
  bodies name dependencies and (if leaked) driver errors. *Mitigation:* token-gate everything
  but `/ping`; `writeError` returns generic messages and logs the detail server-side;
  README's LAN-only + firewall guidance; loopback-default bind.
- **R2 — Shared-WAL multi-process contention.** Collector (writer) and daemon (readers) hit
  the same file. *Mitigation:* WAL allows concurrent readers + one writer across processes;
  `busy_timeout=5000` (already set in `migrate`) absorbs the rare writer-lock wait; the daemon
  never writes rows. Residual: a very long collector transaction could briefly delay a reader —
  negligible at home scale and query sizes.
- **R3 — A stale/misleading `/health/check`.** If `DAEMON_STALE_AFTER` is set below the cron
  period, the daemon reports the collector dead between every run. *Mitigation:* default 90m
  (≈3× the 30-min soft cron); documented as "set to a few × your cron interval".
- **R4 — Token mishandling.** A logged or timing-leaked token defeats the only auth.
  *Mitigation:* `subtle.ConstantTimeCompare`; never log the header; keep it out of error
  strings; `LoadDaemonConfig` never echoes it.
- **R5 — Layering regressions.** `gateway/httpapi` importing `infrastructure`, or an
  `infrastructure→gateway` health assertion, silently inverts the rule. *Mitigation:* the
  `HealthProbe`/`HistoryRepository` ports live in `services`; explicit `go list -deps` checks
  in Tasks 3/8 acceptance criteria; probes are wired in `cmd/yarddogd`, not cross-imported.
- **R6 — The `cmd/` move collides with in-flight 003 work on `main.go`.** *Mitigation:*
  sequence 004 after 003 merges; the move is `git mv` of 003's final files (imports unchanged).
- **R7 — 003 slips.** The `metrics`/`host` endpoints and the exact freshness query depend on
  003's schema. *Mitigation:* the scaffold, `runs`/`checks` endpoints, `sqlite` probe, auth,
  and health pair do not; if 003 slips, ship those and add the metrics/host endpoints when
  003 lands (they are additive to `HistoryRepository` + `Store`).
- **R8 — Unbounded reads.** A missing/oversized `limit` could scan the whole table.
  *Mitigation:* `QueryService` clamps every limit (default + hard max); the store always binds
  a `LIMIT ?`.

## Trade-offs

- **T1 — Second binary under `cmd/` vs a subcommand of one binary.** *Chosen:* two binaries,
  shared layer packages, per-binary composition roots. *Why:* CLAUDE.md mandates it ("a second
  binary gives it its own composition root under `cmd/`; do not build a shared bootstrap"), and
  the two have opposite lifecycles (short-lived cron vs long-running server) and disjoint config
  needs. *Rejected:* one binary with a `serve` subcommand (couples the collector's required
  router/Telegram config to a daemon that needs neither) and per-metric binaries (explicitly
  out of scope).
- **T2 — Inbound HTTP under `gateway/httpapi` vs a new top-level `api/` layer.** *Chosen:*
  `gateway/httpapi`, mirroring the existing outbound `gateway/telegram`/`gateway/router`. *Why:*
  an HTTP server is an external-system boundary — the same layer, just the inbound direction;
  it keeps the six-package shape and the established dto idiom. *Rejected:* a new architectural
  layer (speculative structure for one adapter).
- **T3 — Read port + `QueryService` in `services` vs handlers calling `Store` directly.**
  *Chosen:* a consumer-defined `HistoryRepository` port with a thin `QueryService`. *Why:* keeps
  the dependency rule (handlers never touch SQL), centralizes limit-clamping and not-found
  translation, and gives the handlers a fake-able seam without a DB. *Rejected:* handlers on the
  concrete `Store` (drags `infrastructure` into `gateway`, inverts the layering).
- **T4 — `HealthProbe`/`Inspector` in `services` vs `gateway/httpapi`.** *Chosen:* `services`.
  *Why:* the compile-time `var _ services.HealthProbe = (*Store)(nil)` assertion is legal only
  if the port is where `infrastructure` may import it (`services`), not `gateway`; and readiness
  ("can it do its job") is an application concern. *Rejected:* the inspector in `httpapi` (would
  force an illegal `infrastructure→gateway` assertion or push wiring hacks into `cmd`).
- **T5 — `DAEMON_MAX_CONNS` default 2 vs 1 vs a larger pool.** *Chosen:* default 2,
  operator-tunable. *Why:* the daemon's health probes and its data reads share the pool; at 1,
  a single in-flight `/api/v1/metrics` read holds the only connection and `/health/check`'s
  `sqlite` probe queues behind it, so a merely *busy* daemon reports itself *unhealthy* (a
  false `503`) — the moment a monitor trusts the signal most. Two connections give the health
  path its own headroom; a file DB under WAL takes concurrent readers safely. *Revised during
  review* — this plan originally shipped default 1 on the "serialized reads are imperceptible"
  argument, which missed the probe-vs-data-read contention. *Rejected:* default 1 (the false
  `503` above); a larger pool (WAL-reader concurrency surface for no measured benefit at home
  scale).
- **T6 — Both processes migrate vs a designated migrator.** *Chosen:* both run the idempotent
  `CREATE TABLE IF NOT EXISTS` migration. *Why:* it is safe from either process and removes a
  startup-ordering dependency (the daemon works on a fresh box before the collector's first
  run). *Rejected:* a migration owner (adds ordering coupling for no safety gain).
- **T7 — Daemon exit codes 0/1/2, disjoint from the collector's 0–5.** *Chosen:* a separate,
  documented 3-code contract. *Why:* a server's failure modes (config, listen) do not map onto
  the reboot-centric 3/4/5; reusing `domain.ExitOK`/`ExitConfigError` for 0/1 keeps the common
  cases shared, and `2 = listen error` is the daemon's own. *Rejected:* reusing the collector's
  full table (its 3/4/5 are meaningless for a daemon) or inventing a fresh 0/1 (needless divergence).
- **T8 — `/api/v1` path versioning vs header/none.** *Chosen:* path version. *Why:* trivially
  visible, greppable, and lets a future breaking shape land at `/api/v2` without disturbing
  clients — cheap insurance for a wire contract. *Rejected:* unversioned (a breaking change
  would silently break scrapers) and header-negotiated versions (ceremony for a home API).
