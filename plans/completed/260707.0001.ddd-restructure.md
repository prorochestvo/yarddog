# Task Breakdown

## Overview

Restructure `yarddog` from a single flat `package main` (~2000 lines production, ~2600
lines test, 8 files at repo root) into a Domain-Driven / layered architecture under the
existing module `github.com/prorochestvo/yarddog`, single binary preserved.

This is an **owner-confirmed decision** that deliberately overrides the current
CLAUDE.md rules ("flat by concern", "no `internal`/`pkg`", "no speculative structure",
"no interface until a real consumer"). The migration is not relitigated here; rewriting
those CLAUDE.md sections to describe and mandate the new layout is itself Task 8.

The prime mover is **device-driver extensibility**: the router is one external device
among many to come (design §14 names a Tapo/Sonoff smart-plug hard-reset fallback, and
other Nokia/ALU ONTs already need their own client). The reboot path is promoted to a
driver *family* behind a factory keyed on a new `ROUTER_KIND` config value, so adding a
device is "add a file + a factory case", not "fork `router.go`".

Target layers (owner-specified, bare package names under the one module):
`domain`, `services` (application), `infrastructure`, `gateway`, `gateway/router`,
`gateway/telegram`, `gateway/dto`. Composition root stays a root-level `package main`.

The reference architecture is `com_lingocrm/pkg/authapi` (layered service with
`domain`/`application`/`infrastructure`/`gateway`). We adopt its two load-bearing
idioms — **separation of responsibility by layer** and **consumer-defined port
interfaces** — and reject its HTTP-service-only concepts (inbound `httpV1` handlers/
routes, `sdk`, `swagger`) that a portless CLI has no equivalent for.

### Dependency rule (the invariant every task must preserve)

```
        domain   (imports nothing internal; pure values + pure policy funcs)
          ▲
          │
       services  (imports domain only; DEFINES all port interfaces at the consumer)
          ▲
   ┌──────┴───────┐
infrastructure  gateway/*   (import domain + services; IMPLEMENT the ports)
          ▲       ▲
          └───┬───┘
            main   (composition root; imports every layer; nothing imports main)
```

Arrows point in the allowed import direction. `domain` never imports outward.
`services` imports only `domain` and never touches `infrastructure`/`gateway`.
Adapters (`infrastructure`, `gateway/*`) depend inward on `services` (to name the ports
they satisfy and to host the `var _ Port = (*impl)(nil)` assertions) and on `domain`.
`main` is the only package that may import all of them, and the only one that calls
`os.Exit`, holds the `flock`, and reads the full config.

## Assumptions

- **Ports are EXPORTED interfaces in `services`.** This is a deliberate divergence from
  authapi (whose ports are unexported + satisfied structurally + mirrored in
  `infrastructure/mock`). Justified by: yarddog needs a device *factory* returning a
  common port type, and CLAUDE.md mandates `var _ Port = (*impl)(nil)` assertions living
  next to each impl — both require the port to be nameable from the adapter package.
  Exported ports + inward adapter→services imports is textbook Ports & Adapters; the
  dependency arrow still points inward. See Trade-off 1.
- **`services` production code imports only `domain`.** Adapters importing `services` is
  allowed and expected; `services` importing an adapter is a build-breaking violation.
- **Test code may import inward or sideways freely.** A `services` test that needs a
  real store is out of scope by design (see Task 2 / Risk R2): `services` tests use fakes.
- The completion gate stays `CGO_ENABLED=0 go vet ./... && CGO_ENABLED=0 go test ./...`.
  `./...` already covers subpackages, so **no CI or Makefile change is required**: the
  arm64 build is `go build -o ./build/yarddog-arm64 .` and the root stays `package main`,
  so `PKG := .` is unchanged (confirmed in `.github/workflows/{ci.main,release}.yml` and
  `Makefile`).
- Module Go version `go 1.26.4`; `modernc.org/sqlite` stays the only direct third-party
  dependency (its transitive indirects in `go.mod` are unaffected by a package move).
- No behaviour change is in scope. Every exit code, `runs.outcome`, message template,
  SQL schema, and probe verdict is preserved byte-for-byte. This is a pure move +
  re-seam refactor plus one additive feature (`ROUTER_KIND` + the driver factory).
- `go 1.26` `t.Context()` is already used in tests and stays.

## Finalized decisions on the ambiguous placements

**Config → `infrastructure`; no god-config crosses a layer boundary.**
`Config` + `LoadConfig` + the env loader live in `infrastructure`. The **composition
root is the only consumer of the whole `Config`**; it destructures it into the narrow
constructor arguments each layer already expects (`NewChecker(ips, domains, addr,
timeout)`, `NewNokiaDriver(addr, user, pass, timeout)`, `NewStore(ctx, dbPath)`, a
`services.Settings{Label, RebootCooldown, RecoveryInterval, RecoveryTimeout}`). This is
exactly what `buildDependencies` does today (constructors already take narrow params;
only `Execute` takes the whole `*Config`, which Task 2 narrows to `Settings`). Rejected:
(a) `Config` in `domain` — it is anemic settings, not policy, and would drag env/file
concerns toward the core; (b) a separate top-level `config` package — an extra layer the
owner did not name, buying nothing for a struct only `main` reads whole. See Trade-off 2.

**Telegram adapter → `gateway/telegram` (not `infrastructure`).**
The line between the two adapter layers: **`infrastructure`** = generic/owned technical
capability with no third-party wire contract (SQLite, wall clock, generic TCP/HTTP
connectivity probing, the env). **`gateway/*`** = an adapter to a *specific external
system's bespoke protocol*, each with wire DTOs in `gateway/dto` (the router device
family; the Telegram Bot API). Telegram has a bespoke API + JSON DTO ⇒ `gateway`. This
is why the connectivity **checker stays in `infrastructure`** (it probes anonymous
internet targets with no bespoke protocol/DTO) while **Telegram moves to `gateway`** —
consistent with the owner's seed (checker→infra, telegram→gateway).

**Outbox → split across all three lower layers, cleanly.**
- **Queue persistence** (`tg_outbox` table + its 4 CRUD ops) → `infrastructure/store.go`,
  exposed as the `services.OutboxRepository` port.
- **Reliable-delivery orchestration** (enqueue-before-send, drain-all-oldest-first,
  `[queued HH:MM]` annotation, fail-soft on live-send) → `services/outbox.go`
  (`OutboxService`), an application concern. It implements the `Notifier` port the
  orchestrator consumes.
- **Raw send + secret handling** (POST `sendMessage`, bot-token redaction, DSN parse) →
  `gateway/telegram/telegram.go` (`Client`), implementing the `services.Sender` port.
- **Wire body** (`TelegramSendMessageRequest`) → `gateway/dto/telegram.go`.
- **Row value type** (`OutboxMessage`) → `domain` (it crosses the `OutboxRepository`
  port, so both `services` and `infrastructure` must name it).

**Composition root → root `main.go` (not `cmd/yarddog/`).**
Single binary ⇒ the module *is* the tool. Keeps `Makefile` `PKG := .` and both CI
`go build … .` invocations unchanged, matches Go's norm for single-command modules.
`cmd/` earns its place only when a *second* binary appears; adding it now is the
speculative structure the relaxed rules still don't reward for one binary. If a second
entry point ever lands, introduce `cmd/<name>/main.go` and **inline each startup** — no
shared `bootstrap`/`wiring` package (CLAUDE.md standing rule survives the restructure).

**`gateway/router` is one flat package, one file per device.**
`factory.go` (the `New` switch) + `nokia.go` (the only driver today). Per-device
*sub*packages (`router/nokia/`) are deferred until a driver grows large enough to earn
isolation — matching authapi's "multiple concerns, flat files in one package" and
avoiding a one-driver-per-directory ceremony.

**No `infrastructure/mock` package.**
authapi ships `infrastructure/mock` because its mocks are reused across packages. Every
yarddog fake is used by exactly one test package (`services`), so the fakes live in
`services/fakes_test.go` (unexported, in-package). Extract to a `mock` package only if a
second consumer ever appears. This keeps "no speculative structure" honoured even under
the relaxed regime.

### Seed mappings adjusted (and why)

- **`RunUpdate` struct → `domain`, but its `buildAssignments()` SQL method → `infrastructure`.**
  The struct crosses the `RunRepository.UpdateRun` port so it must be nameable by both
  sides ⇒ `domain`. But SQL-fragment building is persistence detail that must not sit in
  a pure layer ⇒ re-home the method as an unexported `runUpdateAssignments(domain.RunUpdate)`
  helper in `store.go`. Not called out in the seed; flagged here as a domain-purity trap.
- **Quorum verdict extracted into `domain.Verdict`.** The seed *allows* pure policy in
  domain; we *do* it: `allFailed` + the `ip-down OR domain-down` rule become
  `domain.Verdict(targets) domain.Result`, and `infrastructure.Checker.Check` calls it
  after probing. This is the device-agnostic policy the owner wants in the core, and it
  puts the one graded correctness rule under a pure, trivially-tested function.
- **`OutboxMessage` → `domain`** (seed left it unplaced): it crosses `OutboxRepository`,
  so `domain` is its only shared home.
- **`Downtime` given a real home:** a pure `domain.Downtime(rebootStartedAt, restoredAt
  time.Time) time.Duration` free function honours the seed's "downtime concept" without
  inventing a ceremonial type; formatting (`humanDuration`) stays in `services` as
  presentation.

**Nothing in the seed was rejected outright.** The only pushback is scoping ceremony
(no `mock` pkg, no per-device subpackages, no separate `config` pkg, no god-config type)
— documented above and in Trade-offs.

## Tasks

### Task 1: Create `domain` package; move pure values + policy

- **Description:** Create `domain/` and move every pure, I/O-free type, enum, and
  policy function out of the flat package. Update all remaining `package main` references
  to the qualified `domain.X`. No behaviour changes.
  - `domain/check.go`: `Result`, `TargetResult`, `CheckKind{IP,Domain,Gateway}`, and a
    new `Verdict(targets []TargetResult) Result` + unexported `allFailed` (carved from
    `check.go`'s `Check`). `Verdict` computes `Down = allFailed(ip-kind) ||
    allFailed(domain-kind)` from the target kinds.
  - `domain/run.go`: `Run`, `Check`, `RunUpdate` (struct only — **not** its
    `buildAssignments` method), `Mode{Soft,Hard}`, `Action{None,Reboot,SkippedCooldown}`,
    `Phase{Initial,Recovery}`, `Outcome{OK,RebootFailed,Timeout,Skipped}`.
  - `domain/outbox.go`: `OutboxMessage`.
  - `domain/watchdog.go`: `Exit{OK,ConfigError,LockHeld,RebootFailed,RecoveryTimeout,
    SkippedCooldown}`, `reason{NoInternet,ScheduledHardReboot}` (export as
    `ReasonNoInternet` etc. since `services` now consumes them across the package
    boundary), `RouterKind` string type + `RouterKindNokia` + `ParseRouterKind(string)
    (RouterKind, error)` defaulting empty→`nokia`, and `Downtime(from, to time.Time)
    time.Duration`.
  - At this stage `store.go`'s `buildAssignments` becomes a method/func operating on
    `domain.RunUpdate`; the `col*`/`table*` consts stay in `store.go`. Everything except
    `domain/` is still `package main`; `main` imports `domain`.
- **Acceptance Criteria:**
  - [ ] `domain` imports only stdlib (`time`) — `go list -deps ./domain` shows no
        `github.com/prorochestvo/yarddog/{services,infrastructure,gateway}`.
  - [ ] `CGO_ENABLED=0 go vet ./... && CGO_ENABLED=0 go test ./...` green.
  - [ ] `domain/check_test.go` holds the quorum table tests (one failed target ≠ down;
        all IP down = down; IP up + all domain down = down; empty group ≠ down) as
        `TestVerdict` with `t.Run` subtests; latency-recording tests stay with the
        prober in Task 3.
  - [ ] `domain/watchdog_test.go` covers `ParseRouterKind` (empty→nokia, "nokia"→nokia,
        unknown→error) and `Downtime`.
  - [ ] Public-surface-first declaration order within each new file (CLAUDE.md).
  - [ ] Godoc on every exported identifier, name-first, ending in a period.
- **Pitfalls & edge cases:**
  - `CheckKindDomain = "domain"` is a *string value* that happens to equal the package
    name — no conflict, but do not rename the value (it is persisted in `checks.kind`
    and matched in tests).
  - Exporting `reason*` and `Exit*`/`Mode*`/`Outcome*` changes their identifiers
    (`reasonNoInternet` → `domain.ReasonNoInternet`); grep every usage before moving.
  - `Verdict` must key off `TargetResult.Kind`, not slice position, so the caller can
    pass one flat `[]TargetResult` (matches how `Check` builds results today).
  - Do not move `RunUpdate.buildAssignments` — moving the struct without leaving the SQL
    method behind is the domain-purity trap; the method stays in `store.go`.
  - `humanDuration` is presentation → stays with the orchestrator (Task 2), not domain.
- **Complexity:** Medium (wide but mechanical; the `Verdict` carve is the only logic move).
- **Code Example:**
  ```go
  // domain/check.go
  package domain

  // Verdict applies the conservative down rule (design §4.2): the internet is down when
  // every ip target fails OR every domain target fails. A single failed target within a
  // group never trips it; an empty group is "not applicable", never vacuously down.
  func Verdict(targets []TargetResult) Result {
      var ip, dom []TargetResult
      for _, t := range targets {
          switch t.Kind {
          case CheckKindIP:
              ip = append(ip, t)
          case CheckKindDomain:
              dom = append(dom, t)
          }
      }
      return Result{Down: allFailed(ip) || allFailed(dom), Targets: targets}
  }
  ```

### Task 2: Create `services` package; define ports; move orchestrator + OutboxService

- **Description:** Create `services/` importing only `domain`. This is where the flat
  package's `run.go` orchestrator and the outbox half of `telegram.go` land, and where
  every port is defined.
  - `services/ports.go`: the seven interfaces in the port table, plus
    `Settings{Label string; RebootCooldown, RecoveryInterval, RecoveryTimeout
    time.Duration}` (the narrow config slice the orchestrator needs).
  - `services/orchestrator.go`: `runner`, `Execute`, and the full soft/hard flow +
    recovery state machine (`softFlow`, `hardFlow`, `rebootSequence`, `recoveryLoop`,
    `handleGatewayTransition`, `finish*`, `skip*`, `persist*`, `notify`, `logf`,
    `humanDuration`). Change `Execute`'s signature from `(ctx, *Config, *Store, chk, rb,
    nt, clk, mode)` to `(ctx, Settings, RunRepository, Checker, Rebooter, Notifier,
    Clock, mode)`. `runner.store *Store` becomes `runner.repo RunRepository`.
  - `services/outbox.go`: `OutboxService` implementing `Notifier` (`Notify`, `Flush`),
    holding an `OutboxRepository` + a `Sender`. Move `attemptSend`, the enqueue-first
    logic, the drain-all loop, and the `[queued HH:MM]` annotation
    (`queuedTimeFormat`). It calls `sender.Send`; token redaction now happens *inside*
    the `Sender` impl (Task 5), so `OutboxService` records whatever error `Send` returns.
  - At this stage `infrastructure`/`gateway` do not exist yet: `main` still holds the
    concrete `Store`/`Checker`/`Router`/telegram send in `package main` and wires them
    into `services.Execute`/`services.NewOutboxService` (they satisfy the ports
    structurally). `main`'s `Notifier` is now `services.NewOutboxService(store, telegramSender)`.
  - `services/fakes_test.go`: `fakeChecker`, `fakeRebooter`, `fakeNotifier`, `fakeClock`
    (carried from `run_test.go`), plus new `fakeRunRepo`, `fakeOutboxRepo`, `fakeSender`,
    and every `var _ services.Port = (*fake…)(nil)` assertion.
- **Acceptance Criteria:**
  - [ ] `go list -deps ./services` shows **no** `…/infrastructure` and no `…/gateway`.
  - [ ] `services.Execute` and `services.NewOutboxService` compile against ports only.
  - [ ] `orchestrator_test.go` (ex `run_test.go`, all 9 subtests) passes using
        `fakeRunRepo` in place of the real store: happy path message ordering, fast-reboot

  - [ ] `outbox_test.go` (ex `telegram.go`'s outbox subtests) passes with `fakeOutboxRepo`
        + `fakeSender`: enqueue-then-send, live-send failure stays queued + records
        `attempts`/`last_error`, `Flush` drains oldest-first past a `MarkOutboxSent`
        failure (inject `fakeOutboxRepo.markSentErr` for one id — replaces the old
        `CREATE TRIGGER` white-box), `[queued HH:MM]` suffix present.
  - [ ] Every fake carries a compile-time port assertion.
  - [ ] Gate green.
- **Pitfalls & edge cases:**
  - **This is where the white-box `.db` test accesses die** (`run_test.go:119`,
    `telegram_test.go:225`). Both are replaced by fakes returning controlled errors —
    a *better* test seam (they exercise the orchestrator/OutboxService fail-path at the
    port boundary, not via a corrupted DB). The **cooldown query's real SQL correctness**

  - `fakeRunRepo` must faithfully apply non-nil `RunUpdate` pointers onto an in-memory

  - `Execute` no longer sees `*Config`; the two call sites that read `cfg.Label` /

## Execution Order

Strict dependency order; **the gate stays green after every task** (each is its own
commit). Tasks 1→6 each leave a compiling, passing tree because a layer is only peeled
once the packages it depends on exist, and `main` re-wires the still-`package main`
remnants into the new ports at each step.

1. **Task 1 — `domain`.** Leaf; nothing depends on it yet. Green.
2. **Task 2 — `services`** (ports + orchestrator + OutboxService + fakes). Depends on 1.
   `main` wires still-`package main` concretes into the new ports. **This is where the
   white-box `.db` test accesses are replaced by fakes** — do not defer it. Green.
3. **Task 5a — `gateway/dto`** (the dto slice only). Leaf-ish (depends on `domain`); pull
   it *before* Task 4 so the Nokia driver and telegram client consume real wire shapes
   from the start rather than inlining-then-moving. Green.
4. **Task 3 — `infrastructure`** (store/checker/clock/config/env). Depends on 1, 2. `main`
   swaps to `infrastructure.*`. The `:memory:` single-conn store test lands here. Green.
5. **Task 4 — `gateway/router`** (nokia driver + factory + `ROUTER_KIND`). Depends on
   1, 2, 3 (config), 3-of-5a (dto). Green.
6. **Task 5b — `gateway/telegram`** (Client `Send` + finish outbox wiring in `main`).
   Depends on 1, 2, 5a. After this `package main` holds only the composition root. Green.
7. **Task 6 — rewire `main.go`.** Depends on 1–5. Green.
8. **Task 7 — test/fake/assertion reconciliation + full audit** (`go test ./...`,
   `make lint`, opt-in `make test-race`). Depends on 1–6. Green.
9. **Task 8 — CLAUDE.md + README + env.example.** Docs-only; last, so it describes the
   final tree. Green (no code).

**Parallelizable:** none of 1–6 in production code (each depends on the prior peel). The
dto sub-slice (step 3) and CLAUDE.md drafting (Task 8) *can* be drafted in parallel with
coding, but must land in the order above to keep the gate green. If two engineers split
the work, one can pre-write `gateway/dto` + `domain` (steps 1, 3) while the other stages
`services` — but the `services` merge must wait on `domain`.

**Atomic vs staged — recommendation: STAGED, one commit per task.** A Go tree cannot be
half-migrated within a commit (a moved type with un-updated importers does not compile),
but it *can* be migrated across many commits that each compile and pass — and it should
be, for reviewability and `git bisect`. The staging above is chosen specifically so each
commit is green; the alternative single "big-bang" commit is rejected (unreviewable,
un-bisectable, and it would hide the one genuinely tricky step — the Task 2 test re-seam
— inside a 4000-line diff). The one place staging is *forced* to do real design work
rather than a move is Task 2 (the store-port + fake introduction); that is deliberately
early so the rest is mechanical.

## Risks

- **R1 — Import cycle via ports.** If any `services` production file imports
  `infrastructure`/`gateway`, or if a port signature drags an adapter-only type into
  `services`, the build breaks with a cycle. *Mitigation:* ports reference only `domain`
  types; `go list -deps ./services | grep -E 'infrastructure|gateway'` must stay empty
  (add it to the Task 2/3 acceptance checks). The composition root is the only integrator.
- **R2 — Fake/impl divergence in `services` tests.** With `services` tests on fakes
  (never the real store), a fake could drift from the real `Store` semantics and let a
  bug pass. *Mitigation:* keep fakes **dumb** — settable return values +
  pointer-apply, never re-implemented SQL. The behaviours that need real SQL (cooldown
  query, retention, migration, chmod, single-conn) are covered by
  `infrastructure/store_test.go` on `:memory:`. The layering *sharpens* each test's
  responsibility; document this so a reviewer does not read it as lost coverage.
- **R3 — Dropped subtest during the split.** `run.go` and `telegram.go` tests fan out to
  `services` + `gateway`; a subtest could vanish in the move. *Mitigation:* Task 7 asserts
  the total subtest count is preserved (splits are 1→N, never N→0).
- **R4 — Secret-safety regression across the outbox seam.** Redaction moving from
  `Notifier` to the `Sender` impl could leave a path where the raw token reaches the
  persisted `last_error`. *Mitigation:* `Sender.Send` returns an already-redacted error;
  `assertNoRawDSNLeak` runs in `gateway/telegram` tests; add an `outbox_test` assertion
  that a `fakeSender` error containing the token would still be recorded redacted (the
  fake returns a pre-redacted error to mirror the contract).
- **R5 — arm64 cross-compile breaks.** A stray non-buildtag-guarded import could break the
  Pi build. *Mitigation:* `make build-arm` in Tasks 3 and 6 acceptance; CI already runs
  the arm64 build on every push (`ci.main.yml`).
- **R6 — Surviving CLAUDE.md constraint deleted in the rewrite.** *Mitigation:* Task 8
  diffs against the explicit "keep intact" list.
- **R7 — Behaviour drift hidden in a "pure move".** Any accidental change to a message
  template, exit code, or the `recoveryLoop` transition logic is a user-visible
  regression. *Mitigation:* the tests are the guard — they assert exact message ordering,
  exit codes, and persisted timestamps; they must pass unchanged in intent.

## Trade-offs

- **T1 — Exported ports + inward adapter→`services` imports, vs authapi's unexported
  ports + structural satisfaction + `infrastructure/mock` mirror.** Chosen: exported
  ports. *Why:* the device factory needs a shared return type (`services.Rebooter`) and
  CLAUDE.md mandates `var _ Port = (*impl)(nil)` next to each impl — both require naming
  the port from the adapter. authapi's structural style needs the mirror-interface
  ceremony (`tokenRepositoryContract`) precisely because its ports are unexported; at
  yarddog's scale that is more machinery for no gain. The dependency arrow still points
  inward (adapter→application→domain), so this is orthodox hexagonal. *Rejected
  alternative named:* unexported ports — rejected for the factory-return-type problem and
  the assertion-home problem.
- **T2 — Config in `infrastructure` + narrow params, vs a god-`Config` threaded through
  every layer, vs `Config` in `domain`.** Chosen: `infrastructure` owns it, `main`
  destructures into narrow constructor args (+ a small `services.Settings`). *Why:* it
  matches authapi's constructor idiom (specific args, no god-config), keeps `domain` free
  of settings, and respects the dependency rule (a shared `Config` in `infrastructure`
  imported by `services` would be an outward dependency — forbidden). *Rejected:* `Config`
  in `domain` (anemic, non-policy, drags env concerns inward) and a separate `config`
  package (an unrequested layer for a struct only `main` reads whole).
- **T3 — Segregated persistence ports (`RunRepository` + `OutboxRepository`) vs one
  `Store` port.** Chosen: segregated. *Why:* the orchestrator and `OutboxService` consume
  disjoint method sets; ISP keeps each fake tiny and each consumer honest about its
  surface. Cost: two `var _` assertions on the one `infrastructure.Store`. *Rejected:*
  a single 8-method `Store` port — simpler to wire but couples both services to methods
  neither wholly uses, and fattens both fakes.
- **T4 — `gateway/router` as flat files vs per-device subpackages.** Chosen: flat
  (`factory.go` + `nokia.go`). *Why:* one driver exists; subpackages are the speculative
  structure the relaxed regime still shouldn't add pre-emptively. The factory's return
  type already gives the extension seam. *Revisit* when a driver needs its own helper
  types or a vendor SDK — then `gateway/router/<vendor>/` earns isolation.
- **T5 — `gateway/dto` given real Nokia structs vs a telegram-only folder.** Chosen: real
  content for both boundaries (`NokiaLoginForm.Encode()`, reboot body, markers +
  `TelegramSendMessageRequest`) with round-trip tests. *Why:* the owner flagged the
  empty-folder risk; a dto folder holding one struct is ceremony, but making the Nokia
  wire contract explicit and tested is genuinely useful and gives every future driver a
  place for its payloads. Cost: a thin indirection between `nokia.go` and its form
  encoding — accepted as the price of a non-empty, meaningful `dto`.
- **T6 — Root `main.go` vs `cmd/yarddog/main.go`.** Chosen: root. *Why:* single binary,
  unchanged Makefile/CI, Go norm for single-command modules. *Rejected:* `cmd/` — pure
  speculation for one binary; adopt it only when a second entry point exists, and even
  then inline startup per binary (no shared bootstrap), per the surviving CLAUDE.md rule.
