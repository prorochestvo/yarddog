# Task Breakdown

## Overview

Build the full `yarddog` MVP: a short-lived Go CLI, fired by cron on a Raspberry Pi,
that checks the internet uplink and reboots a Nokia G-240W-A GPON router over its web UI
when the link is down. The authoritative specification is
`plans/000-router-watchdog-design.md`; this plan decomposes §16 of that document into
seven ordered build tasks plus their tests.

The design is final — do not re-litigate its decisions (§15 is resolved: TCP/HTTP checks
not ICMP, cron minutes 7/37, cooldown 2h, DSN `tbot://<chat_id>:@<bot_token>/`). Every
acceptance criterion below traces to a design-doc section.

Architecture (design §11, CLAUDE.md): one flat `package main` at the repo root, one file
per concern, no sub-packages, no `internal/`. The only permitted third-party dependency
is `modernc.org/sqlite`. Tests use stdlib `testing` only — no testify. The completion
gate is `go vet ./... && go test ./...` (no `-race` in the default gate; it forces cgo).

The recovery loop is built against four interfaces — `checker`, `rebooter`, `notifier`,
`clock` — so `run.go` is testable with fakes: no real network, no real sleep.

## Assumptions

- The v1 Nokia login/reboot client mentioned in design §14/§16 is **not present** in this
  repo (only scaffolding exists). Task 4 therefore writes `router.go` from the endpoint
  contract described in the design (`login.cgi` → `reboot.cgi`, `sid`/`lsid` cookie,
  `"done reboot"` marker) rather than porting existing code. If a v1 source turns up, the
  engineer reuses it; the acceptance criteria are unchanged either way.
- `reboot.cgi` starts with an **empty request body**; if the router answers "reboot not
  confirmed", the real captured bytes get pasted into a constant (design §14). MVP ships
  the empty-body path and leaves a documented seam.
- Module path is `github.com/prorochestvo/yarddog`; existing `main.go` holds only a
  package doc and an empty `func main` and will be replaced in Task 7.
- Message copy, log lines, times: English, UTC, RFC3339, tag-first `#REBOOT {LABEL} …`
  (design §8.4, §9).
- Config durations (`CHECK_TIMEOUT`, `RECOVERY_INTERVAL`, …) are parsed with
  `time.ParseDuration` (the `.env` examples use `5s`, `15m`, `2h` forms).

## Tasks

### Task 1: config.go + env.go `.env` mini-loader + tests

- **Description:**
  Implement the config surface (design §3) and the ~20-line `KEY=VALUE` loader (design
  §3, no dependency). `env.go` exposes a loader that reads a file into a `map[string]string`:
  skips blank lines and `#`-comment lines, splits on the first `=`, trims surrounding
  quotes from values. `config.go` defines the `Config` struct, a `New`/`Load`-style
  constructor that merges file values with the real process environment (**real env
  wins over file**, design §3), applies defaults, parses durations/ints, and validates.
  Required keys: `LABEL`, `TELEGRAMBOT_DSN`, `ROUTER_USER`, `ROUTER_PASS`. Optional with
  defaults per the §3 table: `ROUTER_ADDR=http://192.168.1.1`,
  `DB_PATH=/var/lib/yarddog/yarddog.db`, `CHECK_IPS=1.1.1.1:443,8.8.8.8:53`,
  `CHECK_DOMAINS=…generate_204,…cdn-cgi/trace`, `CHECK_TIMEOUT=5s`,
  `RECOVERY_INTERVAL=60s`, `RECOVERY_TIMEOUT=15m`, `REBOOT_COOLDOWN=2h`,
  `RETENTION_DAYS=90`. `CHECK_IPS`/`CHECK_DOMAINS` split on comma. Missing required value
  → error that main maps to exit 1.
- **Acceptance Criteria:**
  - [ ] `env.go` loader returns a `map[string]string`; blank lines and `#` lines are
        skipped; `KEY="quoted value"` yields `quoted value` (quotes stripped); a line
        with no `=` is ignored or errors (pick one, document it); a missing file is not
        fatal (loader returns empty map + nil error so the env-only path works).
  - [ ] Real environment variable overrides the same key from the file.
  - [ ] All four required keys absent → validation error naming the missing key(s).
  - [ ] Every optional key falls back to the exact default in design §3 when unset.
  - [ ] Duration fields parse via `time.ParseDuration`; `RETENTION_DAYS` parses as int;
        an unparseable duration/int is a validation error.
  - [ ] `CHECK_IPS`/`CHECK_DOMAINS` parse to `[]string` split on `,` with surrounding
        space trimmed.
  - [ ] `env_test.go` (`TestLoad` or `TestLoadEnv`) covers: quotes, comments, blank
        lines, real-env-wins-over-file, missing file. `config_test.go` covers: all
        required present → ok, each required missing → error, defaults applied, bad
        duration → error. One `Test*` per function, scenarios as `t.Run` subtests.
- **Pitfalls & edge cases:**
  - Real env must win over file — the merge order is the whole point; a test must assert
    it explicitly.
  - Never read or edit a real `.env`; tests write to `t.TempDir()`. `.claude/settings.json`
    denies `.env` access. Edit `yarddog.env.example`, never a live env file.
  - Quote stripping must handle both `"…"` and `'…'` or explicitly only `"…"` — match the
    design's "снятие кавычек"; document which.
  - `RETENTION_DAYS=0` is **valid** and means "keep forever" (design §3/§9) — do not treat
    0 as unset/invalid.
  - Values may contain `=` (bot token, password): split on the **first** `=` only.
- **Complexity:** Easy

### Task 2: store.go — SQLite migrations, tables, cooldown query, retention + `:memory:` tests

- **Description:**
  Implement the persistence layer (design §9) over `modernc.org/sqlite`. On open, set
  `PRAGMA journal_mode=WAL; PRAGMA busy_timeout=5000` and apply the three tables +
  indexes idempotently with `CREATE TABLE IF NOT EXISTS` (design §9). Tables: `runs`,
  `checks`, `tg_outbox` with exactly the columns listed in §9. Provide methods for: insert
  a `runs` row and update its phase timestamps/outcome/error; insert `checks` rows
  (`phase` ∈ `initial`|`recovery`, `kind` ∈ `ip`|`domain`|`gateway`); the **cooldown
  query** (most-recent `runs` row with `action='reboot'` and its `reboot_started_at`, so
  the caller can compare age against `REBOOT_COOLDOWN`, design §6); `tg_outbox`
  enqueue/mark-sent/increment-attempts and a "list unsent" query (for Task 5); and
  **retention** pruning `checks` older than `RETENTION_DAYS` at startup. All
  table/column names referenced through `const` declarations (CLAUDE.md — a schema rename
  must surface at compile time, never as a runtime "no such column"). Times stored
  UTC/RFC3339.
- **Acceptance Criteria:**
  - [ ] Opening the store twice (or calling migrate twice) is idempotent — no error, no
        duplicate-table failure.
  - [ ] `runs` has all §9 columns: `id, started_at, mode, internet_ok, action,
        reboot_started_at, router_down_at, router_up_at, internet_restored_at,
        finished_at, outcome, error`. `internet_ok` is nullable (NULL in hard mode).
  - [ ] `checks` has `id, run_id, ts, phase, target, kind, ok, latency_ms, error` plus
        `idx_checks_run` and `idx_checks_ts`.
  - [ ] `tg_outbox` has `id, created_at, text, sent_at, attempts, last_error`.
  - [ ] Cooldown query returns the latest reboot's `reboot_started_at`; with no prior
        reboot it signals "none" (no row / zero-value + ok flag), never a spurious error.
  - [ ] Retention with `RETENTION_DAYS=N>0` deletes `checks` with `ts` older than
        `now-N days` and keeps newer rows; `RETENTION_DAYS=0` deletes nothing.
  - [ ] Every table/column name used in Go is a `const`; a grep for a raw `"runs"` /
        `"checks"` string literal outside the const block returns nothing.
  - [ ] `store_test.go` on `:memory:` (`TestStore_*`, subtests): idempotent migration,
        insert+read run, insert+read checks, cooldown younger-than-threshold vs
        older-than-threshold vs no-prior-reboot, retention prunes-old/keeps-new, retention
        0 keeps everything, outbox enqueue→list-unsent→mark-sent.
- **Pitfalls & edge cases:**
  - `:memory:` databases are per-connection — if the driver opens a fresh connection the
    schema vanishes. Constrain the pool (`db.SetMaxOpenConns(1)`) or share the connection
    so tests see one consistent in-memory DB.
  - Retention comparison must be on RFC3339 UTC strings; lexicographic compare works only
    because the format is fixed-width — keep all timestamps UTC/RFC3339 (design §9).
  - `RETENTION_DAYS=0` = keep forever — the prune must early-return, not run
    `DELETE … < now-0d` (which would wipe everything).
  - Nullable columns (`internet_ok`, all the `*_at`, `outcome`, `error`) need
    `sql.NullString`/`sql.NullInt64` or pointer scanning, not bare `string`/`int`.
  - No `_`-discarded errors; `rows.Close()` in `defer` is the sanctioned exception. Check
    `rows.Err()` after iteration.
- **Complexity:** Medium

### Task 3: check.go — parallel connectivity probes + quorum verdict + tests

- **Description:**
  Implement connectivity checking (design §4). Two target groups probed concurrently, one
  goroutine per target, each latency recorded (design §4.2). `ip` targets:
  `net.DialTimeout("tcp", host:port, CHECK_TIMEOUT)` — raw link, bypasses DNS. `domain`
  targets: HTTP GET to `generate_204`-style endpoints (expect 204 for generate_204, 200
  for cdn-cgi/trace) under `CHECK_TIMEOUT`. A `gateway` probe (TCP-dial `ROUTER_ADDR`
  host) is used by the recovery loop to tell "router alive" (design §5) — expose it here
  so Task 6 reuses it. The verdict (design §4.2): **down ⇔ (all `ip` targets fail) OR
  (all `domain` targets fail)**. Return per-target results (target, kind, ok, latency,
  error) so `run.go` can persist them into `checks`. This is the concrete `checker`
  interface implementation Task 6 depends on.
- **Acceptance Criteria:**
  - [ ] Probes within a group run concurrently (goroutine per target), results collected
        without data races; each result carries a measured latency.
  - [ ] Verdict is DOWN when **all** `ip` targets fail (domains irrelevant).
  - [ ] Verdict is DOWN when **all** `domain` targets fail even if every `ip` target
        succeeds (dead DNS path still triggers a reboot — design §4.2).
  - [ ] Verdict is UP when at least one `ip` **and** at least one `domain` target
        succeed — a single failed target in a group is never a trigger (design §4.2).
  - [ ] Each probe honors `CHECK_TIMEOUT`; a hung target does not stall the batch beyond
        the timeout.
  - [ ] `domain` probe treats a non-2xx/204 status as a failure with the status recorded
        in the error field.
  - [ ] `check_test.go` (`TestCheck_*` / `TestChecker_*`) using `httptest` servers for
        domains and a real listener (or a known-closed port) for IP targets: one-target-
        down ≠ down, all-ip-down = down, ip-up+all-domains-down = down, latency recorded,
        timeout path. Subtests, `t.Parallel()` where no shared state.
- **Pitfalls & edge cases:**
  - The quorum is `(all ip fail) OR (all domain fail)`, **not** "any fail" and not "all
    targets fail" — get the boolean exactly right or you either reboot on a single hiccup
    or never reboot on dead DNS.
  - Empty group edge case: if `CHECK_IPS` or `CHECK_DOMAINS` is empty, "all failed" is
    vacuously true and would force a permanent DOWN — decide and document (treat empty
    group as "not applicable / not down"). Config defaults are non-empty, but guard it.
  - Reuse one `http.Client` with an explicit `Timeout`; do **not** rely on the default
    client (no timeout → a hung server hangs the whole run). Close response bodies in
    `defer`.
  - Collect goroutine results over a channel or a pre-sized slice indexed by target — do
    not append to a shared slice from multiple goroutines without a mutex.
  - HTTP redirects on `generate_204` should not be followed into a false "up"; a 204 with
    empty body is the success signal.
- **Complexity:** Medium

### Task 4: router.go — Nokia login/reboot HTTP client + `httptest` tests

- **Description:**
  Implement the Nokia G-240W-A reboot client (design §5, §14). Flow: POST `login.cgi`
  with `ROUTER_USER`/`ROUTER_PASS` against `ROUTER_ADDR`, capture the session cookie
  (`sid`/`lsid`), then POST `reboot.cgi` carrying that cookie; success is confirmed by the
  `"done reboot"` marker in the response (design §5). Bad credentials → login fails with a
  distinct error. `reboot.cgi` body starts empty; if the router answers "reboot not
  confirmed", a constant holds the captured bytes (design §14) — leave that constant and
  a comment documenting the seam. This is the `rebooter` interface implementation Task 6
  depends on. Use a shared `http.Client` with a cookie jar (or manual cookie forwarding)
  and an explicit timeout.
- **Acceptance Criteria:**
  - [ ] `login.cgi` POST includes the configured user/pass; a successful response yields a
        session cookie that is carried into the `reboot.cgi` request.
  - [ ] `reboot.cgi` success is detected via the `"done reboot"` marker; a response
        lacking it → error mapped by `run.go` to `outcome='reboot_failed'` / exit 3.
  - [ ] Bad-credentials login response → distinct wrapped error, no reboot attempt.
  - [ ] The session cookie set by `login.cgi` is present on the second (`reboot.cgi`)
        request — asserted by the test server.
  - [ ] `router_test.go` (`TestRouter_*`) with an `httptest.Server`: login sets cookie,
        bad creds rejected, reboot confirmed (`"done reboot"`), reboot not confirmed →
        error, cookie forwarded to second request. Subtests. Fake carries a compile-time
        `var _ rebooter = (*…)(nil)` assertion where a fake stands in.
- **Pitfalls & edge cases:**
  - Do **not** log out — the session dies with the reboot (design §5). Attempting a logout
    against a rebooting router just produces a spurious error.
  - The cookie name may be `sid` or `lsid` — forward whatever `login.cgi` set rather than
    hardcoding one; a cookie jar keyed on the server URL is simplest.
  - `ROUTER_ADDR` is a base URL (`http://192.168.1.1`); build `login.cgi`/`reboot.cgi`
    paths off it without double slashes.
  - Empty reboot body is the documented starting point; the "not confirmed" constant is a
    known future edit, not a bug — wire the seam but don't invent bytes.
  - Explicit client timeout; a router mid-reboot will hang the connection otherwise.
- **Complexity:** Medium

### Task 5: telegram.go — DSN parse + sendMessage + outbox flush + `httptest` tests

- **Description:**
  Implement Telegram messaging (design §8) over raw `net/http` — no bot library. Parse the
  DSN `tbot://{chat_id}:@{bot_token}/` **manually** (design §8.1): `strings.CutPrefix`
  the `tbot://`, `strings.TrimSuffix` the trailing `/`, `strings.Cut` on `:@` into chat id
  and token. `sendMessage`: POST
  `https://api.telegram.org/bot{token}/sendMessage` with JSON
  `{"chat_id":…, "text":…}`, 10s timeout (design §8.2). Implement the **outbox** pattern
  (design §8.3) over `tg_outbox` (Task 2): every message is written to the outbox
  **first**, then a send is attempted; success sets `sent_at`, failure leaves it queued
  and records the error; a `flush` drains unsent rows (called at end of the recovery loop
  and at the start of each run). A late message carries its original queued time:
  `… [queued 04:02]` (design §8.3). This is the `notifier` interface implementation Task 6
  depends on. The base API URL must be overridable so tests point at an `httptest` server.
- **Acceptance Criteria:**
  - [ ] DSN parse: valid `tbot://115818690:@NNN:AAA/` → chat=`115818690`,
        token=`NNN:AAA` (token containing `:` survives — this is why `net/url` is not
        used, design §8.1). Missing `tbot://` prefix → error; missing `:@` separator →
        error; empty token → error; empty chat id → error.
  - [ ] `sendMessage` POSTs to `…/bot{token}/sendMessage` with a JSON body whose
        `chat_id` and `text` match the inputs (asserted by the test server reading the
        request), under a 10s client timeout.
  - [ ] Enqueue-then-send: message is in `tg_outbox` before the HTTP call; on send success
        `sent_at` is set; on send failure the row stays unsent with `attempts`
        incremented and `last_error` recorded.
  - [ ] `flush` sends every unsent row; a row queued earlier is delivered with a
        `[queued HH:MM]` suffix reflecting its `created_at`.
  - [ ] `telegram_test.go` (`TestParseDSN`, `TestSendMessage` / `TestNotifier_*`) with an
        `httptest` server: valid/invalid DSN variants, request shape assertion,
        failure-leaves-in-queue, flush-delivers-with-queued-tag. Subtests; fakes carry
        `var _ notifier = (*…)(nil)`.
- **Pitfalls & edge cases:**
  - **Outbox ordering is the crux** (design §8.3): write to `tg_outbox` **before**
    attempting the send. In soft mode the reboot begins with no internet, so the very
    message that matters most cannot be delivered live — if you send-then-persist, a
    crash loses it. Persist first, always.
  - `net/url` cannot parse the DSN — the bot token legitimately contains `:`. Manual
    `strings.Cut` on `:@` only; never `url.Parse`.
  - `[queued HH:MM]` uses the message's original `created_at` (UTC/RFC3339 stored),
    formatted for humans — not the flush time.
  - The Telegram base URL is `https://api.telegram.org` in prod; make it injectable so
    tests don't hit the real API and don't leak a token.
  - Do not discard the send error — it feeds `last_error`/`attempts` and the outcome.
  - Close the response body; a 10s client timeout is mandatory (a hung API stalls the run).
- **Complexity:** Medium

### Task 6: run.go — orchestrator + recovery state machine over interfaces + fake-clock tests

- **Description:**
  Implement the orchestrator (design §5, §6, §7) and the shared recovery state machine.
  Define the four interfaces (design §11): `checker` (Task 3), `rebooter` (Task 4),
  `notifier` (Task 5), `clock` (now + sleep/after, so tests inject a fake — no real
  sleep). Soft flow (design §6): probe internet, write `runs`+`checks`; if up →
  `action='none'`, `outcome='ok'`, exit 0; if down → consult the cooldown query (Task 2):
  younger than `REBOOT_COOLDOWN` → `action='skipped_cooldown'`, send the cooldown message,
  exit 5; else run the reboot sequence with reason `"no internet"`. Hard flow (design §7):
  no initial check (`internet_ok` NULL), straight to the reboot sequence with reason
  `"scheduled hard reboot"`. Reboot sequence (design §5): flush outbox → send "starting
  router reboot (reason: …)" → `rebooter.Reboot` → recovery loop every
  `RECOVERY_INTERVAL`, max `RECOVERY_TIMEOUT`, each tick probing gateway + internet and
  writing `checks(phase='recovery')`, emitting the phase-transition messages ("router
  went down", "router is up, waiting for internet", final "reboot completed, internet
  restored (downtime …)"), then flush outbox; timeout → "internet still down after 15m,
  giving up", exit 4. Map every terminal state to its (exit code, `runs.outcome`,
  message) triple here — `run.go` is the single place that decides (CLAUDE.md Error
  Handling). Return an exit code to `main.go` rather than calling `os.Exit`.
- **Acceptance Criteria:**
  - [ ] Interfaces `checker`, `rebooter`, `notifier`, `clock` are defined; the recovery
        loop takes them and never calls `time.Sleep` or real network directly.
  - [ ] Soft, internet up: `runs` row `mode='soft'`, `internet_ok=1`, `action='none'`,
        `outcome='ok'`, exit 0. No reboot, no reboot messages.
  - [ ] Soft, internet down, last reboot younger than cooldown: `action='skipped_cooldown'`,
        `outcome='skipped'`, cooldown message sent with the real age ("last reboot 40m
        ago"), exit 5, **no** reboot.
  - [ ] Soft, internet down, no recent reboot: full reboot sequence, reason `"no internet"`.
  - [ ] Hard: no initial check (`internet_ok` NULL), reboot sequence with reason
        `"scheduled hard reboot"`.
  - [ ] Happy recovery path emits messages in order: starting → router went down → router
        is up → reboot completed (with `downtime` = `internet_restored_at −
        reboot_started_at`), outcome `ok`, exit 0.
  - [ ] **Fast reboot** where the DOWN phase is never observed between polls: down/up
        messages are **skipped**, only "starting" + final "reboot completed" are sent
        (design §5).
  - [ ] Recovery timeout: after `RECOVERY_TIMEOUT` with internet still down → "internet
        still down after 15m, giving up", `outcome='timeout'`, exit 4.
  - [ ] `rebooter.Reboot` error → "reboot failed: <error>", `outcome='reboot_failed'`,
        exit 3.
  - [ ] `runs` phase timestamps populated: `reboot_started_at`, `router_down_at`,
        `router_up_at`, `internet_restored_at`, `finished_at`.
  - [ ] `run_test.go` (`TestRun_*` / state-machine test) on injected fake clock + fake
        checker/rebooter/notifier: happy path + message order, fast-reboot-skips-down/up,
        timeout→exit 4, cooldown→exit 5, reboot-failed→exit 3, soft-up→exit 0. Fakes each
        carry compile-time `var _ checker = (*fakeChecker)(nil)` (and peers) assertions.
- **Pitfalls & edge cases:**
  - **Downtime is `reboot_started_at → internet_restored_at`** (design §5), not
    router_down→router_up and not wall-clock of the whole run. Compute it from the two
    stored timestamps.
  - The fast-reboot case is a real branch, not a curiosity: if the router cycles between
    two 60s polls, the loop never sees gateway-down, so down/up transitions must be
    suppressed and only the final message sent. Drive it with the fake clock in a test.
  - Transition detection is edge-triggered (was-alive→now-dead), not level-triggered —
    track previous gateway state across ticks, emit once per transition.
  - The `clock` must abstract both "now" and "wait `RECOVERY_INTERVAL`"; a fake advances
    virtual time so the 15m timeout test runs instantly. No real `time.Sleep` anywhere in
    the state machine.
  - Cooldown age string ("last reboot 40m ago") is derived from the cooldown query's
    timestamp vs `clock.Now()` — keep it consistent with the injected clock in tests.
  - `run.go` owns the (exit code, outcome, message) mapping — lower layers return plain
    wrapped `%w` errors and let `run.go` decide (CLAUDE.md). Do not scatter `os.Exit`.
  - Flush the outbox at the **end** of the recovery loop (internet is back) so queued
    "starting"/"down"/"up" messages finally deliver (design §8.3).
- **Complexity:** Hard

### Task 7: main.go — CLI parsing, flock, mode selection, exit-code mapping

- **Description:**
  Replace the scaffold `main.go` with the entry point (design §2, §10). Parse flags:
  bare invocation = soft mode, `--hard-reboot` = hard mode, `--env <path>` = env file path
  (default `/etc/yarddog.env`, fallback `./.env`, design §2/§3). Acquire an exclusive
  non-blocking `flock` on `/var/run/yarddog.lock` (`syscall.Flock`, `LOCK_EX|LOCK_NB`,
  design §10.1); if held → log + exit 2. Load config (Task 1) → on error log + exit 1.
  Open the store (Task 2), run startup retention + a startup outbox flush (design §8.3),
  construct the real `checker`/`rebooter`/`notifier`/`clock`, dispatch to `run.go`'s
  orchestrator, and `os.Exit` with the returned code. `main` is the **only** place that
  calls `os.Exit`. Exit codes exactly per design §2: 0 ok, 1 config error, 2 lock held,
  3 reboot failed, 4 recovery timeout, 5 cooldown skip.
- **Acceptance Criteria:**
  - [ ] Bare `yarddog` → soft mode; `yarddog --hard-reboot` → hard mode; `--env <path>`
        overrides the config path; default path `/etc/yarddog.env` with `./.env` fallback.
  - [ ] `flock` on `/var/run/yarddog.lock` is `LOCK_EX|LOCK_NB`; a second concurrent
        instance exits 2 without touching the router.
  - [ ] The lock fd is held for the whole process lifetime (including the up-to-15m
        recovery loop) and released only on exit — not closed early.
  - [ ] Config load failure → exit 1; all other terminal codes (0/3/4/5) come from
        `run.go`'s returned code, mapped straight to `os.Exit`.
  - [ ] Startup performs retention prune and an outbox flush before dispatching.
  - [ ] `os.Exit` appears only in `main`; `run.go` and lower layers return errors/codes.
- **Pitfalls & edge cases:**
  - **flock is released by the kernel on process death — a stale lock is impossible**
    (design §10.1). Do **not** build lockfile-liveness/PID-staleness logic; that
    reintroduces the bug flock avoids. Just `LOCK_EX|LOCK_NB` and exit 2 on contention.
  - Keep the lock file descriptor open for the process lifetime; if it's closed (e.g. a
    `defer f.Close()` that fires too early, or GC of the `*os.File`) the lock drops mid-
    recovery and the 04:00 hard run can overlap. Hold the fd in a variable that outlives
    the run.
  - `/var/run` may be `/run`; the path is a config-independent constant per design §10 —
    ensure the process (root, via cron) can create/open it.
  - `os.Exit` skips deferred `Close()` calls — flush/close the store and release resources
    before exiting, or accept that WAL checkpoints on next open (document the choice).
  - The cron collision at 04:00 is handled by minutes 7/37 **and** flock (design §10);
    main only owns the flock half. Do not encode cron scheduling in the binary.
- **Complexity:** Medium

## Execution Order

Strict dependency chain, mirroring design §16:

1. **Task 1 (config + env)** — foundation; everything reads `Config`.
2. **Task 2 (store)** — depended on by outbox (5), cooldown/checks persistence (6),
   retention (7).
3. **Task 3 (check)**, **Task 4 (router)**, **Task 5 (telegram)** — **parallelizable**:
   each is an independent leaf implementing one of the interfaces the orchestrator
   consumes. Task 5 needs Task 2 (outbox tables); Tasks 3 and 4 depend only on Task 1.
4. **Task 6 (run — orchestrator/state machine)** — depends on 2, 3, 4, 5 (defines and
   consumes all four interfaces).
5. **Task 7 (main — CLI/flock/exit)** — depends on 1, 2, 6; wires the real
   implementations and owns `os.Exit`.

After each task and before review: `go vet ./... && go test ./...` must be green
(the completion gate; no `-race` — it forces cgo, off the pure-Go build path).

## Risks

- **Nokia endpoint contract is inferred, not captured** — `login.cgi`/`reboot.cgi` field
  names, the exact `"done reboot"` marker, and the reboot body may differ on the live
  firmware. Mitigation: httptest tests pin the *client's* behavior; the empty-body →
  captured-bytes seam (design §14) absorbs the "reboot not confirmed" case; the first real
  smoke run (design §13.5) validates against hardware. A firmware update changing endpoints
  surfaces as `reboot_failed` in Telegram, not a silent failure.
- **`:memory:` connection-pool trap** — a multi-connection pool loses the schema between
  calls. Mitigation: single-connection store in tests (Task 2 pitfall).
- **Provider outage vs router fault** — a reboot cannot fix an upstream outage; without
  cooldown the tool would reboot every 30 min through the whole outage. Closed by the
  cooldown gate (design §6, §14), exercised in Task 6 tests.
- **Message loss when it matters most** — the Pi's only uplink is through the router being
  rebooted. Mitigation: outbox-first persistence (Task 5) + flush on recovery and at next
  run start. A regression here is invisible until a real outage; the enqueue-before-send
  test guards it.
- **Instance overlap** — a 15m recovery loop can meet the next cron tick, and `*/30` fires
  at 04:00 alongside the hard reboot. Closed by flock + minutes 7/37 (design §10); the
  flock fd must be held for the whole process (Task 7 pitfall).
- **Power loss mid-recovery** — leaves a `runs` row without `finished_at`; acceptable and
  visible in the DB (design §14). No mitigation needed beyond not treating it as corruption.

## Trade-offs

- **TCP/HTTP probes over ICMP** — chose TCP-dial + HTTP GET (design §4.1) over real ICMP.
  Why: raw ICMP sockets need root or `cap_net_raw` and add complexity for no functional
  gain here; the TCP+DNS path is equivalent for "is the uplink usable" and keeps the
  binary privilege-free and dependency-free. Cost: doesn't detect pure ICMP-only
  reachability, which is irrelevant to this router-reboot use case.
- **Flat `package main`, no `internal/`** — chose one flat package split by concern
  (design §11, CLAUDE.md). Why: for a single-binary utility this size, packages/layers are
  over-engineering; the four interfaces earn their place solely as test seams. Cost: no
  compiler-enforced layer boundaries — discipline is by file, not by package.
- **SQLite as the message queue (outbox)** — reused the same SQLite file as the Telegram
  queue rather than a separate mechanism. Why: the DB is already the durable store, and the
  outbox needs exactly its durability + at-startup-flush semantics; a second queue system
  would be pure overhead. Cost: couples message delivery to DB availability — acceptable,
  since a dead DB means the whole run is already broken.
- **flock + shifted cron minutes (belt and suspenders)** — both mechanisms guard overlap
  though either alone mostly suffices (design §10). Why: minutes 7/37 remove the designed
  04:00 collision, and flock covers the *emergent* overlap (a long recovery meeting the
  next tick) that scheduling can't. Cost: two things to keep in sync (crontab + binary),
  but the binary owns only the flock half.
- **`modernc.org/sqlite` (pure Go) over cgo `mattn/go-sqlite3`** — chose the cgo-free
  driver (design §9, CLAUDE.md). Why: it cross-compiles Mac/x86 → linux/arm64 with no C
  toolchain, keeping `CGO_ENABLED=0` on every build. Cost: a slightly slower driver and
  the `-race` gate needs a separate cgo-enabled box (`make test-race`) — accepted; the
  default gate stays pure Go.
