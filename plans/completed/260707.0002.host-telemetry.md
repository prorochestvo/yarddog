# Task Breakdown

## Overview

Add cross-platform **host telemetry** to `yarddog`: each cron invocation records one
**snapshot** of host metrics into SQLite alongside the connectivity check, then exits.
No daemon, no ports — the per-run snapshot preserves yarddog's short-lived/portless
premise. Two independent operator toggles ship with it:

- **`METRICS_ENABLED`** (master) + a per-collector toggle each — pick exactly what a host
  records.
- **`REBOOT_ENABLED`** — turn the router reboot off for a **monitor-only** deployment
  (still checks, records, notifies; never touches the router).

The collector set: **temperature** (°C, per sensor), **fans** (RPM, best-effort),
**cpu** (loadavg 1/5/15 — *not* CPU%), **memory** (total/free/available), **disk** (usage
for a configured mount), **uptime**, **network** (per-iface rx/tx byte counters). Each is
individually toggleable; a collector unsupported on the running platform is recorded as an
**unavailable sample** (`ok=0` + reason), never an error.

### Hard constraints (owner re-confirmed mid-design — non-negotiable)

- **`CGO_ENABLED=0` on every target, including darwin.** The binary must cross-compile to
  `linux/arm64`, `linux/amd64`, **and** `darwin/arm64` with **no C toolchain**.
- **The only permitted third-party dependency stays `modernc.org/sqlite`.** No
  `golang.org/x/sys` (it exists in `go.mod` *only* as a transitive indirect of
  `modernc.org/sqlite` — metrics code must never `import` it), no `gopsutil`, no SMC/IOKit
  shim, no subprocess that needs a C library. This is *why* the macOS temperature/fan path
  is dropped (SMC needs cgo) — do not reintroduce it by any route.
- Every metric read is pure-Go stdlib: `os.ReadFile` of sysfs/procfs, `syscall.Statfs`
  (stdlib, both GOOS), and `os/exec` of stock macOS binaries (`sysctl`, `vm_stat`) with
  **no sudo**. `syscall.Statfs` and `os/exec` need neither cgo nor `golang.org/x/sys`.

### Targets (confirmed)

pi5 (Ubuntu Server `linux/arm64`), a generic Ubuntu Server `linux/amd64`, and a MacBook M1
on **macOS** (`darwin/arm64`). macOS still collects the portable subset (cpu/memory/uptime/
disk); temperature, fans, and network are recorded unavailable there.

### Grounded facts from the live pi5 (probed — do not re-theorize)

- `uname`: `aarch64`, Ubuntu 24.x `…-raspi`. **`vcgencmd` does NOT work** (`/dev/vcio`
  missing) — read sysfs, never depend on the Raspberry Pi userland.
- Temperature works via sysfs: `/sys/class/thermal/thermal_zone0` (`type=cpu-thermal`,
  `temp=52350` millidegrees → 52.35 °C); `/sys/class/hwmon/hwmon0` (`name=cpu_thermal`,
  `temp1_input=52350`); `/sys/class/hwmon/hwmon2` (`name=rp1_adc`, `temp1_input=54310`).
- **No fan RPM on this box**: zero `fan*_input` anywhere under `/sys`. Fans are therefore
  best-effort — present only when a cooler's hwmon driver exposes `fanN_input`.
- procfs staples present: `/proc/loadavg`, `/proc/meminfo`, `/proc/uptime`, `/proc/net/dev`.

### Fit with the current layered architecture

The dependency rule is `domain ← services ← {infrastructure, gateway} ← main`. New code
lands as: value types + vocabulary in `domain/`; two consumer-defined ports + the
orchestrator step + the reboot gate in `services/`; the SQLite tables + the build-tagged
platform collectors + the config toggles in `infrastructure/`; wiring in `main.go`. No new
package, no second binary, no shared bootstrap layer (CLAUDE.md Code Organization
Principles).

## Assumptions

- **`REBOOT_ENABLED` defaults `true`.** Any other default would silently turn every already
  deployed watchdog into monitor-only on upgrade — a dangerous behaviour change. Non-
  negotiable backward-compat requirement.
- **`METRICS_ENABLED` and every per-collector toggle default `true`.** The feature exists to
  collect telemetry; all-on gives the richest out-of-the-box data, and unsupported
  collectors self-document as `ok=0` rows. Existing installs start writing metrics on the
  next run — acceptable: a new table, low volume (~15-20 rows/run), pruned by retention.
- **One snapshot per run**, taken once right after the initial connectivity check (soft) or
  at run start (hard). The up-to-15 min recovery loop does **not** take extra snapshots.
- **`RebootEnabled` / `MetricsEnabled` are `bool` fields on `services.Settings`, not `domain`
  types.** `domain` owns the *vocabulary* (the new `outcome`/`action`/`collector` string
  consts + the metric value types); a bare toggle belongs with the other orchestrator
  settings (`Label`, `RecoveryTimeout`, …). This is a deliberate, documented refinement of
  the task's "`RebootEnabled` flag in `domain`" wording — see Trade-off T1.
- **Metrics are best-effort and orthogonal to the core job.** A collector never returns an
  error (unavailable = a row), and a `SaveMetrics` failure is logged and swallowed — it can
  **never** change a run's `outcome` or exit code (mirrors `persistCheck`'s log-and-continue).
- **The pure parsers are the tested surface.** Every string→value parser (linux *and*
  darwin) lives in a **no-build-tag** file so the Linux CI runner compiles and tests them,
  including the darwin ones it would otherwise never see. Only thin OS-access wiring is
  build-tagged; the darwin wiring is covered by a `GOOS=darwin` compile-guard, not by tests.
- **Darwin network is recorded unavailable** for now (procfs is linux-only; `netstat -ib`
  parsing is a clearly-scoped follow-up). This keeps darwin's `os/exec` surface to exactly
  the `sysctl`/`vm_stat` the owner specified.
- Retention reuses **`RETENTION_DAYS`**: the `metrics` and `host` rows are pruned by `ts` at
  startup exactly like `checks`. `runs` rows are never pruned (unchanged).
- Module path `github.com/prorochestvo/yarddog`, `go 1.26.4`; completion gate stays
  `CGO_ENABLED=0 go vet ./... && CGO_ENABLED=0 go test ./...` (the Linux gate never compiles
  `metrics_darwin.go` — hence the compile-guard, Task 8).

## Data model

Long / EAV table for the runtime-toggleable, variable-cardinality metric set, plus a
per-run `host` sidecar for the string-valued identity that does not fit a numeric EAV row.

```sql
CREATE TABLE IF NOT EXISTS metrics (
  id        INTEGER PRIMARY KEY,
  run_id    INTEGER NOT NULL REFERENCES runs(id),
  ts        TEXT    NOT NULL,          -- snapshot time, UTC RFC3339 (one ts per snapshot)
  collector TEXT    NOT NULL,          -- 'temperature'|'fans'|'cpu'|'memory'|'disk'|'uptime'|'network'
  name      TEXT    NOT NULL,          -- measurement key: 'cpu-thermal','load1','/ used_ratio','eth0 rx_bytes',...
  value     REAL,                      -- NULL when the sample is unavailable/unsupported/failed
  unit      TEXT    NOT NULL,          -- 'celsius'|'rpm'|'bytes'|'ratio'|'load'|'seconds'|''
  ok        INTEGER NOT NULL,          -- 1 available, 0 unavailable
  error     TEXT                       -- reason when ok=0 ('unsupported on darwin', 'no fan sensors present', read error)
);
CREATE INDEX IF NOT EXISTS idx_metrics_run ON metrics(run_id);
CREATE INDEX IF NOT EXISTS idx_metrics_ts  ON metrics(ts);

CREATE TABLE IF NOT EXISTS host (
  run_id   INTEGER PRIMARY KEY REFERENCES runs(id),   -- exactly one host row per run
  ts       TEXT    NOT NULL,
  hostname TEXT    NOT NULL,           -- os.Hostname()
  os       TEXT    NOT NULL,           -- runtime.GOOS
  arch     TEXT    NOT NULL            -- runtime.GOARCH
);
```

**Why EAV over wide columns.** The enabled collector set is runtime-toggleable and the
sensor cardinality is unknown at compile time and per-host (pi5 exposes `cpu-thermal` +
`cpu_thermal` + `rp1_adc` and zero fans; an amd64 box exposes `coretemp` per-core + fans;
the Mac exposes none). Wide columns cannot express a variable, per-host set, and would need
a schema change per new sensor. EAV also expresses "unavailable" uniformly (one row,
`value` NULL, `error` explains). It is exactly the shape `checks` already uses (one row per
probed target) — consistent with the codebase, not a new idiom. Cost: aggregate queries
filter on `collector`/`name` instead of selecting a column — acceptable for a low-volume
local diagnostic DB.

**Why a `host` sidecar over columns on `runs`.** Host identity is per-run-stable **string**
data that does not fit the numeric `value REAL` EAV row. Putting it on `runs` would require
`ALTER TABLE runs ADD COLUMN` on the live pi5 DB — but the store migrates with
`CREATE TABLE IF NOT EXISTS` only (no column-add path). A sidecar keyed by `run_id` keeps
`runs`' schema frozen (no migration framework needed) and matches the existing "sidecar
tables reference `runs(id)`" shape (`checks`). See Trade-off T2.

**`value REAL` precision.** REAL (float64) holds integers exactly to 2^53 (~9 PB); memory,
disk, and rx/tx byte counters on these hosts stay far below that within a counter's
lifetime. Uniform `value REAL` beats a mixed-type column.

## Environment variables (authoritative table lives in README — Task 9)

All booleans parsed by `strconv.ParseBool` (`true/false/1/0/t/f`, case-insensitive); an
unparseable value fails `LoadConfig` (exit 1, config error) naming the key. All optional.

| Variable | Default | Semantics |
|---|---|---|
| `REBOOT_ENABLED` | `true` | Master reboot switch. `false` = **monitor-only**: soft/hard flow checks, records the snapshot, notifies, but never calls the router; records `outcome=reboot_disabled`, exit `0`. |
| `METRICS_ENABLED` | `true` | Master telemetry switch. `false` = no snapshot at all (no `host` row, no `metrics` rows). |
| `METRICS_TEMPERATURE` | `true` | Per-sensor temperature (°C). Unsupported on darwin → `ok=0` row. |
| `METRICS_FANS` | `true` | Fan RPM. **Best-effort**: no `fanN_input` present → one `ok=0` "no fan sensors present" row. Unsupported on darwin. |
| `METRICS_CPU` | `true` | Load averages 1/5/15. **Not CPU%** — a one-shot cron process cannot sample utilisation over an interval; loadavg needs no interval. |
| `METRICS_MEMORY` | `true` | Total / free / available memory (bytes). |
| `METRICS_DISK` | `true` | Usage of `METRICS_DISK_MOUNT`: total/free/avail bytes + used ratio. |
| `METRICS_UPTIME` | `true` | Host uptime (seconds). |
| `METRICS_NETWORK` | `true` | Per-interface rx/tx **byte counters** (cumulative since boot, not a rate — diff two runs for throughput). Unavailable on darwin (pending `netstat -ib`). |
| `METRICS_DISK_MOUNT` | `/` | Filesystem mount the disk collector stats (single mount; CSV multi-mount is a future extension). |

**Per-collector toggle vs unsupported.** A **disabled** collector (`METRICS_X=false`)
produces **no rows** — the operator turned it off. An **enabled-but-unsupported / absent /
failed** collector produces `ok=0` row(s) with an `error` — the gap is visible. This is why
the collector must know its enabled set (see `domain.MetricsSettings`).

## Reboot-disabled terminal semantics (chosen)

New vocabulary (in `domain`): `ActionSkippedDisabled = "skipped_disabled"` (parallel to the
existing `skipped_cooldown`) and `OutcomeRebootDisabled = "reboot_disabled"`.

| Scenario | `REBOOT_ENABLED` | Internet | `runs.action` | `runs.outcome` | Exit | Telegram |
|---|---|---|---|---|---|---|
| soft, up | `false` | up | `none` | `ok` | `0` | *(none — identical to a normal up run)* |
| soft, down | `false` | down | `skipped_disabled` | `reboot_disabled` | `0` | `#REBOOT {LABEL} no internet, reboot disabled (monitor-only)` |
| hard | `false` | (n/a) | `skipped_disabled` | `reboot_disabled` | `0` | `#REBOOT {LABEL} reboot disabled (monitor-only): skipping scheduled hard reboot` |
| any | `true` | any | *(unchanged)* | *(unchanged)* | *(unchanged)* | *(unchanged)* |

**Exit `0` is deliberate.** Exit codes `3/4/5` are all about the reboot *attempt* (failed /
timed out / cooldown-skipped). Monitor-only disables the reboot path entirely, so none
apply — the process succeeded at its configured (monitor-only) job. The outage is surfaced
via Telegram **and** the `reboot_disabled` outcome in the DB; the exit code stays `0` so the
machine/cron contract remains "did the reboot path fail?" A dedicated exit `6` is *rejected*
(Trade-off T3): it would add a code for a state the DB/Telegram already expose and muddy the
contract. **soft-up with `REBOOT_ENABLED=false` is byte-for-byte a normal up run** — reboot-
disabled only diverges when a reboot *would* have happened.

**Noise note (accepted, flagged):** soft monitor-only re-notifies every down run (~2/hour
during an outage), matching the existing cooldown-skip behaviour. Per-run dedup would need
last-run state (yarddog is stateless per invocation) — out of scope; see Risk R6.

## Build-tag file layout & the pure/impure split

```
infrastructure/
├── metrics.go              # NO build tag — the PURE surface + host identity.
│                           #   compiles & is unit-tested on EVERY OS (incl. Linux CI):
│                           #   parseLoadavg, parseMeminfo, parseUptime, parseNetDev,
│                           #   parseThermalTemp, parseHwmonTemp, parseHwmonFan,          (linux parsers)
│                           #   parseSysctlLoadavg, parseSysctlUint, parseSysctlBoottime, (darwin parsers)
│                           #   parseVmStat, diskUsage, uptimeSeconds, darwinAvailableBytes,
│                           #   hostInfo(), unavailable() helper.
├── metrics_test.go         # NO build tag — tests ALL parsers (linux + darwin) via captured fixtures.
├── metrics_linux.go        # //go:build linux — linuxCollector: sysfs/procfs reads under an
│                           #   INJECTABLE root, syscall.Statfs; delegates ALL parsing to metrics.go.
│                           #   defines NewMetricsCollector; var _ services.MetricsCollector assertion.
├── metrics_linux_test.go   # //go:build linux — fixture-tree tests (runs on the Linux CI runner).
├── metrics_darwin.go       # //go:build darwin — darwinCollector: os/exec sysctl/vm_stat,
│                           #   syscall.Statfs; temp/fans/network → unavailable. defines NewMetricsCollector.
└── testdata/metrics/
    ├── pi5/{sys,proc}/...   # replicates the live pi5 tree: thermal_zone0=cpu-thermal, hwmon0=cpu_thermal,
    │                        #   hwmon2=rp1_adc, NO fan*_input; /proc/{loadavg,meminfo,uptime,net/dev}.
    ├── amd64/{sys,proc}/... # synthetic: a hwmon WITH fan1_input=1200 + coretemp — exercises the fan-present path.
    └── empty/{sys,proc}/... # no sensors/procfs — exercises the all-unavailable path.
```

**The split, and why it is the whole testability story:**

| Layer | Lives in | Tested where | Runs on Linux CI? |
|---|---|---|---|
| Pure parse (string→value), **linux + darwin** | `metrics.go` (no tag) | `metrics_test.go` (no tag) | **Yes** — incl. darwin parsers |
| Linux OS-access (file reads under `root`, Statfs) | `metrics_linux.go` | `metrics_linux_test.go` (fixtures) | **Yes** — real reads, no hardware |
| Darwin OS-access (`os/exec`, Statfs) | `metrics_darwin.go` | *(none — cannot exec darwin on Linux)* | **No** — guarded by `GOOS=darwin` build (Task 8) |

The trick: darwin **parsers** live in the no-tag file, so the CI runner compiles and tests
the logic where bugs actually hide (`parseVmStat`, `parseSysctlBoottime`, …). Only the
darwin **exec/syscall wiring** ships untested-on-CI, and that is caught at *compile* time by
the `CGO_ENABLED=0 GOOS=darwin GOARCH=arm64 go build ./...` guard. A package-level unexported
parser that is only *called* from a build-tagged file still compiles on the other OS (Go
does not flag unused package-level funcs) — and `metrics_test.go` calls it, so it is neither
dead nor untested.

### Per-platform collector matrix & the CGO=0 / no-dep guarantee

| Collector | linux mechanism | darwin mechanism | cgo? | dep? |
|---|---|---|---|---|
| temperature | `os.ReadFile` sysfs `thermal_zone*/temp`, `hwmon*/temp*_input` | **unavailable** (SMC needs cgo) | no | none |
| fans | sysfs `hwmon*/fan*_input` (best-effort) | **unavailable** (SMC needs cgo) | no | none |
| cpu (loadavg) | `os.ReadFile` `/proc/loadavg` | `exec sysctl -n vm.loadavg` | no | none |
| memory | `os.ReadFile` `/proc/meminfo` | `exec sysctl -n hw.memsize` + `exec vm_stat` | no | none |
| disk | `syscall.Statfs` | `syscall.Statfs` | no | none |
| uptime | `os.ReadFile` `/proc/uptime` | `exec sysctl -n kern.boottime` (uptime = now − boottime) | no | none |
| network | `os.ReadFile` `/proc/net/dev` | **unavailable** (procfs is linux-only) | no | none |
| host identity | `os.Hostname` + `runtime.GOOS`/`GOARCH` (portable, in `metrics.go`) | same | no | none |

`syscall.Statfs` field types differ per GOOS (`Bsize` is `int64` on linux, `uint32` on
darwin), so the `Statfs` call + field extraction live in the platform files; the byte math
(`total = blocks*bsize`, `used = total-free`) is the pure shared `diskUsage(...)`. Darwin
`exec` calls are bounded by `exec.CommandContext` with a short (~3 s) timeout so a hung
`sysctl` cannot stall the run. None of `syscall.Statfs`, `os/exec`, `runtime`, `os.ReadFile`,
`bufio`, `strings`, or `strconv` needs cgo or `golang.org/x/sys`.

## DDD placement (respecting `domain ← services ← {infra,gateway} ← main`)

- **`domain/metrics.go`** (imports stdlib only): `Collector` string enum (7 consts),
  `MetricSample{Collector,Name,Value,Unit,OK,Error}`, `HostInfo{Hostname,OS,Arch}`,
  `HostMetrics{Host,Samples}`, `MetricsSettings{7 bools}` + `Enabled(Collector) bool`.
  **`domain/run.go`**: add `ActionSkippedDisabled`, `OutcomeRebootDisabled` consts.
- **`services/ports.go`**: `MetricsCollector interface { Collect(ctx) domain.HostMetrics }`
  and `MetricsRepository interface { SaveMetrics(ctx, runID int64, ts time.Time, m
  domain.HostMetrics) error }` (consumer-defined, reference `domain` only). `Settings` gains
  `RebootEnabled bool` and `MetricsEnabled bool`.
  **`services/orchestrator.go`**: the `collectMetrics` step + the `REBOOT_ENABLED` gate +
  `finishRebootDisabled` terminal. `Execute` gains `metricsRepo MetricsRepository` and `mc
  MetricsCollector` params; `runner` gains the matching fields.
- **`infrastructure/`**: `metrics.go` + `metrics_linux.go` + `metrics_darwin.go` (the build-
  tagged `MetricsCollector` impls sharing the pure parsers); `store.go` gains the `metrics`/
  `host` tables, `SaveMetrics` (`MetricsRepository` impl, one transaction), off-port
  `PruneMetrics`, and the `col*`/`table*`/`idx*` consts; `config.go` gains the toggles.
- **`main.go`**: build the platform collector via `infrastructure.NewMetricsCollector(...)`
  (the build tag picks the impl), pass `st` as `MetricsRepository`, thread the enabled set +
  the two master toggles, call `st.PruneMetrics` at startup.

`PruneMetrics` stays **off-port** on the concrete `*infrastructure.Store` (main calls it,
like `PruneChecks`) — `MetricsRepository` is only what the orchestrator calls (`SaveMetrics`),
keeping ISP.

## Tasks

### Task 1: `domain` — metric value types + reboot/metrics vocabulary

- **Description:** Add `domain/metrics.go` with the pure metric types and `domain/metrics_test.go`;
  extend `domain/run.go`'s action/outcome vocabulary. No I/O, no consumer yet.
  - `domain/metrics.go`: `Collector` (`string`) with `CollectorTemperature/Fans/CPU/Memory/
    Disk/Uptime/Network`; `MetricSample`, `HostInfo`, `HostMetrics`, `MetricsSettings` +
    `Enabled(Collector) bool`. Value type shapes exactly onto a `metrics` row.
  - `domain/run.go`: add `ActionSkippedDisabled = "skipped_disabled"` to the Action block and
    `OutcomeRebootDisabled = "reboot_disabled"` to the Outcome block (godoc updated to name
    the new values, per the existing block comments).
- **Acceptance Criteria:**
  - [ ] `go list -deps ./domain` shows no internal package (stdlib only).
  - [ ] `MetricSample` has exactly `Collector, Name string→Collector/string, Value float64,
        Unit string, OK bool, Error string`; `HostMetrics{Host HostInfo; Samples
        []MetricSample}`.
  - [ ] `domain/metrics_test.go` has `TestMetricsSettings_Enabled` (each kind on/off; an
        unknown `Collector` → false) as `t.Run` subtests.
  - [ ] Public-surface-first order; godoc on every exported identifier, name-first, period-
        terminated.
  - [ ] `CGO_ENABLED=0 go vet ./... && CGO_ENABLED=0 go test ./...` green.
- **Pitfalls & edge cases:**
  - `Value float64 + OK bool` (not `*float64`): "unavailable" is `OK=false` and the store maps
    it to SQL NULL. Keep `OK` and `Error` as the availability signal, not the zero value.
  - `Collector` const **values** (`"temperature"`, `"cpu"`, …) are persisted in
    `metrics.collector` — do not rename them later without a data migration.
  - `MetricsSettings` holds only the 7 enable bools — **not** `DiskMount` (that is a config
    string threaded to the collector constructor separately).
- **Complexity:** Easy.
- **Code Example:**
  ```go
  // domain/metrics.go
  type Collector string
  const (
      CollectorTemperature Collector = "temperature"
      CollectorFans        Collector = "fans"
      CollectorCPU         Collector = "cpu"
      CollectorMemory      Collector = "memory"
      CollectorDisk        Collector = "disk"
      CollectorUptime      Collector = "uptime"
      CollectorNetwork     Collector = "network"
  )

  // MetricSample is one measured or unavailable host metric, shaped to map onto a
  // metrics row. Value is meaningful only when OK; an unavailable, unsupported, or
  // failed sample sets OK=false, leaves Value zero (persisted as SQL NULL), and
  // explains itself in Error.
  type MetricSample struct {
      Collector Collector
      Name      string  // 'cpu-thermal', 'load1', '/ used_ratio', 'eth0 rx_bytes'
      Value     float64
      Unit      string  // 'celsius','rpm','bytes','ratio','load','seconds',''
      OK        bool
      Error     string
  }

  type HostInfo struct{ Hostname, OS, Arch string }
  type HostMetrics struct {
      Host    HostInfo
      Samples []MetricSample
  }
  ```

### Task 2: `REBOOT_ENABLED` monitor-only slice (services gate + config + main)

- **Description:** Ship the reboot toggle end-to-end. It needs **no** collector, so it is a
  clean early vertical slice.
  - `services/ports.go`: add `RebootEnabled bool` to `Settings`.
  - `services/orchestrator.go`: add `finishRebootDisabled(ctx, runID, msg)` (notify + flush +
    `UpdateRun{Action: skipped_disabled, Outcome: reboot_disabled, FinishedAt}` → return
    `domain.ExitOK`). Gate in `softFlow` (when internet down and `!RebootEnabled`, before the
    cooldown query) and `hardFlow` (when `!RebootEnabled`, instead of `rebootSequence`).
  - `infrastructure/config.go`: add `RebootEnabled bool` to `Config`; parse `REBOOT_ENABLED`
    (default `true`) via a new `getBool` helper (reused by Task 4).
  - `main.go`: set `services.Settings{… RebootEnabled: cfg.RebootEnabled}`.
  - Tests: `orchestrator_test.go` subtests (reuse existing fakes) + `config_test.go` subtest.
- **Acceptance Criteria:**
  - [ ] `orchestrator_test.go`: `soft down + RebootEnabled=false` → `fakeRebooter.calls == 0`,
        run `outcome=reboot_disabled` / `action=skipped_disabled`, exit `domain.ExitOK`, one
        message `"no internet, reboot disabled (monitor-only)"`; `hard + RebootEnabled=false`
        → `fakeRebooter.calls == 0`, same outcome, message `"reboot disabled (monitor-only):
        skipping scheduled hard reboot"`; `soft up + RebootEnabled=false` → identical to a
        normal up run (`action=none`, `outcome=ok`, no message).
  - [ ] `RebootEnabled=true` leaves every existing test unchanged (behaviour byte-for-byte).
  - [ ] `config_test.go`: `REBOOT_ENABLED` default true; `=false` parsed; `=notabool` → error.
  - [ ] Gate green.
- **Pitfalls & edge cases:**
  - `finishRebootDisabled` still calls `notifier.Flush` (a hard-mode monitor run may have live
    internet and queued messages to drain; a soft-down flush is a harmless no-op attempt).
  - The gate in `softFlow` goes **before** the cooldown query (`GetLastRebootStartedAt`) — a
    monitor-only host must not care about cooldown, and must never call the rebooter.
  - Do not add a new exit code — reboot-disabled is `ExitOK` (see the semantics table).
  - `getBool` on an empty/absent value returns the default (mirror `getOrDefault`'s "empty ==
    unset" rule so `REBOOT_ENABLED=` does not blank the default to false).
- **Complexity:** Medium.
- **Code Example:**
  ```go
  // services/orchestrator.go
  func (r *runner) finishRebootDisabled(ctx context.Context, runID int64, msg string) int {
      r.notify(ctx, msg)
      if err := r.notifier.Flush(ctx); err != nil {
          r.logf("flush outbox: %v", err)
      }
      action, outcome := domain.ActionSkippedDisabled, domain.OutcomeRebootDisabled
      finishedAt := r.clock.Now()
      if err := r.repo.UpdateRun(ctx, runID, domain.RunUpdate{
          Action: &action, Outcome: &outcome, FinishedAt: &finishedAt,
      }); err != nil {
          r.logf("update run %d: %v", runID, err)
      }
      return domain.ExitOK
  }

  // in softFlow, replacing the "internet down" fall-through into cooldown:
  if !r.settings.RebootEnabled {
      return r.finishRebootDisabled(ctx, runID, "no internet, reboot disabled (monitor-only)")
  }
  // ... existing GetLastRebootStartedAt / cooldown / rebootSequence ...

  // infrastructure/config.go
  func getBool(get func(string) (string, bool), key string, def bool) (bool, error) {
      raw, ok := get(key)
      if !ok || raw == "" {
          return def, nil
      }
      v, err := strconv.ParseBool(strings.TrimSpace(raw))
      if err != nil {
          return false, fmt.Errorf("invalid %s %q: %w", key, raw, err)
      }

  func (s *Store) SaveMetrics(ctx context.Context, runID int64, ts time.Time, m domain.HostMetrics) error {
      tx, err := s.db.BeginTx(ctx, nil)
      if err != nil {
          return fmt.Errorf("save metrics for run %d: begin: %w", runID, err)
      }
      defer func() { _ = tx.Rollback() }() // no-op after Commit (sanctioned discard)

      if _, err := tx.ExecContext(ctx,
          fmt.Sprintf(`INSERT INTO %s (%s,%s,%s,%s,%s) VALUES (?,?,?,?,?)`,
              tableHost, colHostRunID, colHostTS, colHostHostname, colHostOS, colHostArch),
          runID, formatTime(ts), m.Host.Hostname, m.Host.OS, m.Host.Arch,
      ); err != nil {
          return fmt.Errorf("save metrics for run %d: host: %w", runID, err)
      }

      for _, sm := range m.Samples {
          if _, err := tx.ExecContext(ctx,
              fmt.Sprintf(`INSERT INTO %s (%s,%s,%s,%s,%s,%s,%s,%s) VALUES (?,?,?,?,?,?,?,?)`,
                  tableMetrics, colMetricsRunID, colMetricsTS, colMetricsCollector, colMetricsName,
                  colMetricsValue, colMetricsUnit, colMetricsOK, colMetricsError),
              runID, formatTime(ts), string(sm.Collector), sm.Name,
              nullFloat(sm.Value, sm.OK), sm.Unit, boolToInt(sm.OK), nullString(sm.Error),
          ); err != nil {
              return fmt.Errorf("save metrics for run %d: sample %s/%s: %w", runID, sm.Collector, sm.Name, err)
          }
      }
      return tx.Commit()
  }

  func nullFloat(v float64, ok bool) any {
      if !ok {
          return nil
      }
      return v
  }
  ```

        never a panic.
  - [ ] `parseThermalTemp("52350")` → `52.35`; `parseLoadavg("0.52 0.58 0.59 1/234 5678")` →
        `(0.52,0.58,0.59)`; `parseUptime("12345.67 2345.6")` → `12345.67`; `parseNetDev`

## Risks

- **R1 — Telemetry breaks the watchdog's core job.** A collector panic or a `SaveMetrics`
  error must never change an `outcome`/exit code. *Mitigation:* `Collect` returns no error
  (unavailable = rows); `SaveMetrics` errors are logged and swallowed (Task 7 test asserts the
  outcome is unchanged under `fakeMetricsRepo.err`); every parser returns an error instead of
  panicking on malformed input (Task 5 tests malformed input).
- **R2 — A darwin-only break ships silently.** `go test`/`go vet` on the Linux runner never
  compile `metrics_darwin.go`. *Mitigation:* the pure darwin parsers are tested on CI (Task 5);
  the `GOOS=darwin go build ./...` guard (Task 8) catches wiring/compile breaks in both CI
  workflows. Residual: a darwin **runtime** bug in the exec/Statfs wiring is caught only by
  manual smoke on the Mac — accepted, flagged in CLAUDE.md.
- **R3 — An accidental cgo / `golang.org/x/sys` dependency.** A "convenient" metrics library
  or an SMC shim would silently break the pure-Go cross-compile. *Mitigation:* `make lint`
  already greps `go.mod`; extend the CLAUDE.md constraint text (Task 9) to name
  `golang.org/x/sys`/`gopsutil`; the `CGO_ENABLED=0 GOOS=darwin` guard fails if cgo creeps in.
  Note `golang.org/x/sys` stays a transitive **indirect** of `modernc.org/sqlite` — that is
  fine; a **direct** import is the violation.
- **R4 — EAV name/value drift.** A renamed `collector`/`name`/`unit` string silently orphans
  historical rows (no FK on those strings). *Mitigation:* `collector` values are the
  `domain.Collector` consts (compile-checked); document the `name`/`unit` scheme in CLAUDE.md;
  they are diagnostic, not queried by the app.
- **R5 — `host.run_id` PK conflict.** A second `SaveMetrics` for one run errors on the PK.
  *Mitigation:* `collectMetrics` is called exactly once per run (Task 7 test asserts one save
  per terminal path); the PK is a correctness guard, not a case to `INSERT OR REPLACE`.
- **R6 — Monitor-only notification noise.** Soft monitor-only re-notifies every down run during
  an outage. *Mitigation:* documented as matching the existing cooldown-skip behaviour;
  per-run dedup (last-run state) is a deliberate non-goal (yarddog is stateless per run).
- **R7 — Statfs field-type divergence.** `Bsize` is `int64` (linux) vs `uint32` (darwin);
  sharing the `Statfs` call would not compile cross-platform. *Mitigation:* extract fields in
  each platform file, convert to `uint64`, then call the shared pure `diskUsage`.

## Trade-offs

- **T1 — `RebootEnabled`/`MetricsEnabled` on `services.Settings`, not a `domain` type.** The
  task listed a "`RebootEnabled` flag in `domain`". *Chosen:* the toggles are `bool` fields on
  `Settings` (with `Label`, `RecoveryTimeout`, …); `domain` owns the *vocabulary* (the new
  `outcome`/`action`/`collector` consts + the metric value types). *Why:* a bare boolean
  toggle is not domain policy — it is orchestrator configuration, and the existing `Settings`
  is exactly where run-shaping toggles live. A `domain.RebootPolicy` wrapper for one bool is
  ceremony. The *decisions* the toggle drives (which outcome/action/exit) do live in `domain`
  + `services`. *Rejected:* a `domain` bool type (over-engineering).
- **T2 — `host` sidecar table vs columns on `runs`.** *Chosen:* a `host(run_id PK, …)`
  sidecar. *Why:* host identity is per-run string data that does not fit the numeric EAV
  `metrics` row, and columns on `runs` would need `ALTER TABLE` on the live DB — but the store
  migrates with `CREATE TABLE IF NOT EXISTS` only. A sidecar freezes `runs`' schema and matches
  the existing `checks`-references-`runs` shape. *Rejected:* `runs` columns (needs a migration
  framework); a `text_value` column on `metrics` (muddies the clean numeric `value REAL`; every
  numeric row would carry an always-NULL text column).
- **T3 — Reboot-disabled is exit `0`, not a new exit `6`.** *Chosen:* `ExitOK`. *Why:* exit
  codes 3/4/5 describe the reboot *attempt*; monitor-only disables the reboot path, so the
  process succeeded at its configured job. The outage is surfaced via Telegram + the
  `reboot_disabled` outcome. *Rejected:* exit `6` (adds a machine code for a state the DB/
  Telegram already expose and complicates the cron contract).
- **T4 — Add two positional params to `Execute` (10 total) vs a `Deps` struct.** *Chosen:* two
  more params (`metricsRepo`, `mc`), matching the flat positional style the restructure just
  settled and how the outbox already passes `st` twice. *Why:* a `Deps` struct would reopen a
  just-decided signature for a feature — scope creep the "no speculative refactor" rule
  discourages. *Revisit:* if a further collaborator lands, a `Deps` struct becomes worthwhile;
  flagged now, not done now.
- **T5 — EAV `metrics` table vs wide columns.** *Chosen:* long/EAV (one row per sample).
  *Why:* the enabled set is runtime-toggleable and sensor cardinality is unknown/per-host;
  wide columns cannot express a variable set and would need a schema change per sensor. EAV
  also expresses "unavailable" uniformly and matches `checks`. *Rejected:* wide columns
  (impossible for dynamic cardinality); JSON blob (non-idiomatic for this SQLite-first store).
- **T6 — Darwin network unavailable now vs `netstat -ib` parsing.** *Chosen:* record
  unavailable; keep darwin's `exec` surface to `sysctl`/`vm_stat`. *Why:* keeps this plan in
  the owner-specified darwin scope; the row stays uniform (`ok=0` + reason). *Revisit:* a
  `parseNetstatIb` pure parser + one `exec` is a small, clearly-scoped follow-up.
- **T7 — One snapshot per run vs sampling through recovery.** *Chosen:* one snapshot, taken
  after the initial check. *Why:* matches "a per-run snapshot" and keeps volume low and the
  data model simple (`host.run_id` PK). *Revisit:* an end-of-recovery second snapshot could
  capture post-reboot host state — deferred, not needed now.
- **T8 — Metrics pruned by `RETENTION_DAYS` (shared) vs a separate `METRICS_RETENTION_DAYS`.**
  *Chosen:* reuse `RETENTION_DAYS`, pruning `metrics`/`host` by `ts` like `checks`. *Why:*
  metrics volume is comparable to checks; one retention knob is simpler and the rows age
  together. *Rejected:* a parallel knob (config surface for no real need); it can be split
  later if metrics volume ever diverges from checks.
